## 1. 目的と位置づけ

### 1.1 目的

- バトル勝敗に連動する **経済ループ**（獲得・消費・ロスト・復帰）を定義する。
- 創作（ピクセル絵・カード編集）とバトル継続の **モチベーション設計** を明文化する。
- 無償プレイ・広告・サブスク・単品課金の **役割分担** を固定し、実装・バランス調整の基準とする。

### 1.2 本書とプロトタイプ仕様の関係

| ドキュメント | 役割 |
|--------------|------|
| [PROTOTYPE_DEVELOPMENT_SPEC.md](./PROTOTYPE_DEVELOPMENT_SPEC.md) | 戦闘ルール・UI・技術構成 |
| **本書（ECONOMY_SPEC.md）** | 通貨・アイテム・Lost・ショップ・課金（**プロトタイプでも正**） |
| [ECONOMY_ROADMAP.md](./ECONOMY_ROADMAP.md) | 段階的実装順序・フェーズ定義 |

- 戦闘エンジン・墓地・撃破ルールはプロトタイプ仕様に従う。
- **敗北時の Lost**（§4.2, §6）はプロトタイプ実装済み。旧「仮ロスト」は廃止（[§17](#17-プロトタイプからの移行)）。

### 1.3 スコープ

本書は **初版（CPU 戦）から将来機能まで** を章として含む。各機能に **実装段階**（`v1` / `将来`）を付記する。

## 2. 設計原則

1. **デッキは常に最大5枚** — ロスト・削除後は5枚揃うまでバトル不可。
2. **同時未処理ロストは最大1枚** — 1枚ロストでバトル不可のため、未処理ロストが2枚以上になることはない。
3. **バトル報酬だけでは主要アイテムを賄えない** — 不足分は広告視聴・サブスク・ジュエル購入で補う。
4. **創作の努力が経済に反映される** — 塗り数・色数が復活コスト・初期レア抽選・戦利品 px に影響する。
5. **操作地点での直消費** — 削除・デッキ3以降・属性セレクト等は **💎 ジュエルをその場で消費**。リネーム・リタッチ・キャンバス拡大は **px**（創作系: レベル **300** / リタッチ **300** / リネーム **200**）。中間 consumable（溶解リキッド等）は使わない。
6. **限界突破は専用資源** — px・💎 では代替不可。**属性かけら**（バトル戦利品）と **汎用かけら**（L≡4 レベルアップ報酬）の二系統。
7. **用語の区別** — 通貨「ピクセル」と、カードの「塗りピクセル数（マス数）」は UI 上で混同しない（[§16](#16-ui--ux-要件)）。
8. **序盤の学習期間** — ユーザーレベル **5 未満** は敗北ロストを発生させない（[§4.2.1](#421-初心者保護ユーザーレベル--5)）。**Lv5 到達** と同時に Lost を解禁し、初回 **護符 ×1** を配布する（[§7.3](#73-取得)、[§10.4](#104-レベルアップ報酬)）。

## 3. 通貨

### 3.1 種類

| 種別 | 名称（UI） | 主な獲得源 | 主な用途 |
|------|------------|------------|----------|
| 無償 | **ピクセル**（px） | バトル勝利、レベルアップ、**カード削除返還**、勝利2倍広告 | **復活**、**護符購入（px のみ）**、**汎用かけら購入**、**リネーム**、**属性リタッチ**、編集時キャンバス拡大 |
| 有償 | **💎 ジュエル** | 課金、**毎レベル 30**、サブスク | **カード削除**・**属性セレクト**・デッキ3以降・限界突破レア昇格・創作拡張の **その場消費** |
| 専用 | **属性かけら** | 勝利時の戦利品選択（§4.1） | 同一属性カードの限界突破（**N=10 / R=15 / SR・UR=20**） |
| 汎用 | **汎用かけら** | **L ≡ 4 (mod 5), L ≥ 5** のレベルアップ（§10.4） | 任意属性のかけらとして消費（属性かけらと**同価値**） |
| 消費品 | **護符** | **Lv5 到達（初回 ×1）**、**Lv20,30,40,50**、ショップ、サブスク | ロスト1回免れ（装備消費） |

- px と 💎 はヘッダー等で **アイコンを分けて** 常時表示する。
- 💎 が必要な操作にはボタンに **💎 と数値** を併記する（中間チケット名は使わない）。
- 属性かけらは **所持品タブ** に一覧表示。マイデッキ詳細の限界突破は [§16.9](#169-限界突破ui) 参照。
- ジュエルは **返金不可の仮想通貨** として扱う（実装・ストア規約は別途）。

### 3.2 廃止したモデル

以下は **採用しない**（2026-06 設計改定）:

- 有償ピクセル / 無償→有償の消費順
- 復活ポーション・溶解リキッド・リネームチケット・デッキ解放キー等の **中間 consumable 商品**

### 3.3 上限

- px・💎 の所持上限は **未定**（実装時に決定。上限なしも選択可）。
- 復活等の px 消費は **整数**（`Math.floor`）。カードごとに塗り・レア・★で変動（§18.1）。

## 4. バトル報酬・ペナルティ

### 4.1 勝利報酬（`v1`: CPU 戦）

| 項目 | 内容 |
|------|------|
| トリガー | プレイヤー勝利 |
| 操作 | 相手 **墓地** から **プレイヤーが1枚選択** |
| 獲得 | **無償ピクセル** ＋ **選択カード属性のかけら**（レア度別・[§9.3](#93-限界突破)） |
| カードデータ | 自デッキには **保存しない**（相手側参照のみ） |
| 墓地が空 | **起こらない**（勝利＝相手が1枚以上撃破済み） |

**生存報酬（確定）**

- 生存枚数 × `PIXELS_PER_SURVIVOR`（プロトタイプ: **10 px/枚**）。

**戦利品 px（プロトタイプ実装）**

```
px = floor(√塗り数 × K × 色係数)
```

- 実装: `calcGraveyardPixelReward`（`src/config/economy.ts`）。

**戦利品 px（将来・レアボーナス案）**

```
獲得 = BASE_WIN_CURRENCY + RARITY_BONUS[選択カードのレア度]
```

| 定数 | 値 |
|------|-----|
| `BASE_WIN_CURRENCY` | 10 |
| N ボーナス | +10 |
| R ボーナス | +15 |
| SR ボーナス | +20 |

**戦利品・属性かけら**

| 項目 | 内容 |
|------|------|
| 基本 | 選択カードの `attribute` に対応するかけら（N=1, R=2, SR=3） |
| レアボーナス | 上記で確定（`GRAVEYARD_SHARD_REWARD`） |
| 限界突破消費 | **かけら合計**（専用＋汎用、1:1。px・💎不可）。**N=10 / R=15 / SR・UR=20** |

- 相手墓地のレア度は **バトル中の `BattleUnit.rarity`** を正とし、戦利品モーダルへ引き継ぐ（`resolveGraveyardLootCards`）。
- バトル準備・対戦中は R/SR+ カードを **枠色＋背景色** で表示（`BattleCard`）。
- マイデッキ詳細では、**かけらが10個以上揃い突破可能なときだけ**、編集・削除ボタンの直上に限界突破 UI を表示（不足時は非表示）。
- 勝利後、**任意**でリワード広告 → 当該戦の **EXP・px・属性かけら ×2**（[§11.1](#111-勝利後2倍)）。プレミアム会員は **常時2倍**（CM なし）。
- 戦利品選択 UI: `GraveyardPickModal` — px とかけらをカードごとに表示。

### 4.1.1 バトル開始広告（非会員・CPU 戦）

| 項目 | 内容 |
|------|------|
| 対象 | 会員未加入の **通常 CPU 戦** 開始 |
| 条件 | **ユーザーレベル ≥ 5**（`LOST_MIN_USER_LEVEL` と同値） |
| 頻度 | **3回に1回**（`(normalBattleStarts + 1) % 3 === 0`。開始前 `MockRewardAdModal`） |
| 会員 | ライト / プレミアム: **解除**（[§12.2](#122-特典と月次配布)） |
| 対象外 | **バトル履歴再戦**（[§4.1.2](#412-バトル履歴再戦v1-プロトタイプ)）— 別カウント `historyRematchStarts`（§11.4） |

- ~~**1日10戦 cap**~~ は **2026-06-20 設計改定で廃止**（未実装）。コード上の `BATTLE_DAILY_FREE_LIMIT` は **レガシー定数**（削除予定）。

### 4.1.2 バトル履歴再戦（`v1` プロトタイプ）

**実装段階: `v1`（プロトタイプ）**

| 項目 | 内容 |
|------|------|
| 入口 | **バトル**ハブ右上 **📊** → **戦績** → **バトル履歴** → 対戦詳細 →「もう一度対戦する」（[PROTOTYPE §4.6](./PROTOTYPE_DEVELOPMENT_SPEC.md#46-戦績バトル履歴)） |
| 初回告知 | **再戦時のルール** モーダル（`HistoryRematchRulesModal`）。「今日はこれ以降表示しない」→ JST 日次キー `historyRematchRulesDismissedDayKey` |
| 広告 | **3回に1回**、ルールモーダル後・**デッキ選択の前** にモックリワード広告（[§11.4](#114-履歴再戦モック)） |
| デッキ選択 | 通常 CPU 戦と同一 UI（`BattleDeckSelectScreen`）。**5枚揃い** なら **ロストカード含め** 出撃可（`deckReadinessMode="historyRematch"`） |
| 相手デッキ | 履歴の **カード構成は維持**。**BP のみ** 現在のユーザーレベルで再計算（`prepareHistoryOpponentDeck`） |
| マッチング演出 | **相手探索（matching）はスキップ**（相手確定済み）。**3秒 reveal カウントダウンは実施**（敵5枚は即スロット表示）。キャンセル **px 不要** |
| 履歴追記 | **しない** |
| 戦績 | カード `wins` / `losses`・ユーザー戦績は **更新しない** |
| EXP | **加算しない** |
| 敗北 | **ロスト抽選なし**（`LostRoulette` スキップ） |
| 勝利報酬 | **生存カード分の px のみ**（墓地選択・属性かけら・勝利2倍広告は対象外）。バトル終了 UI の中央ガイダンス帯に **+N** をインライン表示 |
| 勝敗後 |「もう一度対戦」→ 再びデッキ選択フロー（ルールモーダル・広告判定を含む） |

- UI フロー詳細は [PROTOTYPE §4.6](./PROTOTYPE_DEVELOPMENT_SPEC.md#46-戦績バトル履歴)・[§11.4](./PROTOTYPE_DEVELOPMENT_SPEC.md#114-バトル履歴再戦) 参照。

### 4.2 敗北ペナルティ（`v1` / 将来対人）

| 項目 | 内容 |
|------|------|
| トリガー | プレイヤー敗北 |
| 対象 | 自軍 **墓地** から1枚 |
| CPU 戦 | **ランダム** に1枚選定 |
| 対人戦 | **相手ユーザー** が墓地から1枚選択（[§13](#13-対人戦将来)） |
| 自軍墓地が空 | **起こらない**（敗北＝自軍が1枚以上撃破済み） |
| 結果 | 選ばれた1枚が **Lost** 状態へ（護符があれば [§7](#7-護符)） |
| 前提 | **ユーザーレベル ≥ 5**（未満は [§4.2.1](#421-初心者保護ユーザーレベル--5)） |

### 4.2.1 初心者保護（ユーザーレベル &lt; 5）

**実装段階: `v1`**

| 項目 | 内容 |
|------|------|
| 条件 | **ユーザーレベル &lt; 5** の間 |
| 敗北時 | **ロスト抽選を行わない** — 自軍墓地からの1枚選定・`LostRoulette` 演出・`status: lost` 化をすべてスキップ |
| 戦績 | 敗北として `losses` は加算してよい（[PROTOTYPE §11.3](./PROTOTYPE_DEVELOPMENT_SPEC.md#113-戦績) 準拠） |
| 対象 | **CPU 戦・練習戦** を問わない |
| 解禁 | **Lv5 到達時** から §4.2 の通常ルールを適用。同時に [§7.3](#73-取得) の初回護符を配布し、レベルアップ UI で Lost 解禁を告知する（[§16.8](#168-レベル5到達時lost-解禁)） |
| 既存セーブ | Lv5 以上のユーザーには **遡及しない**（新規および Lv5 未満のプレイのみ対象） |

### 4.3 戦績

- 勝利 / 敗北時、出撃した5枚の `wins` / `losses` を従来どおり更新する（プロトタイプ仕様 §11.3 準拠）。
- **バトル履歴再戦**（[§4.1.2](#412-バトル履歴再戦v1-プロトタイプ)）では **更新しない**。

### 4.4 EXP

- EXP 加算ルールは **プロトタイプ仕様のまま**（経済実装と独立）。
- **バトル履歴再戦**では EXP を **加算しない**（[§4.1.2](#412-バトル履歴再戦v1-プロトタイプ)）。

## 5. 墓地と戦闘終了処理

### 5.1 墓地のスコープ

- **その戦闘中に撃破され墓地へ送られたカードのみ** が対象。
- 戦闘をまたいだ墓地の持ち越しは **ない**。

### 5.2 戦闘終了後のカード状態

| カード | 戦後状態 |
|--------|----------|
| ロスト対象に選ばれた1枚 | `status: lost` |
| 護符で免れた1枚 | `status: active`（護符消滅） |
| その他の自軍5枚 | `status: active`（通常どおりデッキで使用可） |

- 勝利時: 自軍5枚はすべて `active`。相手墓地から通貨化した1枚は相手側データのみ参照（自デッキ変更なし）。
- **ユーザーレベル &lt; 5** で敗北した場合: 自軍5枚はすべて `active` のまま（§4.2.1）。

### 5.3 演出要件

敗北時、**どのカードがロスト対象になったか** を明示する。

| 対戦 | 演出 |
|------|------|
| CPU | 墓地から選ぶ **スロット／ルーレット風** 演出 |
| 対人 | 相手が墓地から選んだカードが **相手操作として可視** |

## 6. Lost（ロスト）状態と復帰

### 6.1 Lost の定義

- カードデータは **削除しない**（Lost プール相当）。
- `status: lost` のカードは **バトル・編集不可**（編集可否は [§20](#20-未決定要調整)）。
- アクティブが5枚未満の間は **バトル開始不可** — これ自体がペナルティの一部。

### 6.2 処理 UI

| 項目 | 内容 |
|------|------|
| タイミング | 敗北直後は **次バトル不可**。 **マイデッキ画面** で処理 |
| 表示 | ロストカードに **ロスト状態の見た目** ＋ 選択ボタン |

### 6.3 復帰オプション

ロストカードごとに、以下から **1つ** を選ぶ（復活上限未到達時）。**降格復活は廃止**（2026-06-20）。

#### 6.3.1 復活

> **プロトタイプ**: 無償 px を **カードごと** に消費（塗り×3×レア×★）。`calcReviveCost(card)`。

| 項目 | 内容 |
|------|------|
| 手段 | **無償ピクセル** をその場で消費 |
| `reviveCount` | **+1**（カードに **「復活 n/3」** と表示） |
| 復活上限 | **`REVIVE_CAP = 3`** |
| 護符 | ロスト免れ時は **`reviveCount` を増やさない** |
| レア度 / BP / ★ | **すべて維持** |
| UI | ボタン表記 **「復活」** のみ（完全復活・降格復活の区別なし） |

#### 6.3.2 削除（全レア・active / lost 共通）

> **プロトタイプ**: **💎 5** 消費 ＋ **px・属性かけら返還**（返還 px は勝利戦利品と同式）。詳細は [§18.1](#181-プロトタイプ簡略実装現行コード)。

| 項目 | 内容 |
|------|------|
| 手段 | 削除ボタン（**💎 消費**） |
| データ | カード **完全削除**（デッキから除去） |
| 報酬 | **px**（`calcGraveyardPixelReward`）＋ **属性かけら**（N=1, R=2, SR=3） |
| 確認 | **二段階**（返還内容を 💎・px・かけらアイコン付きで表示） |
| その後 | **新規カード作成が必須**（5枚揃えるまでバトル不可） |

#### 6.3.3 思い出アルバムに保存（全レア・active / lost 共通）

> **プロトタイプ**: **無償**。デッキから外し **閲覧専用アーカイブ** へ移す。詳細は [§14.1](#141-思い出アルバムプロトタイプ実装済)。

| 項目 | 内容 |
|------|------|
| 手段 | **思い出アルバムに保存** ボタン |
| コスト | **なし**（px・💎 消費なし。**返還もなし**） |
| データ | デッキから **除去** → `memoryAlbum.cards` に追加（`status: active` で保存） |
| 確認 | **1段階**（「デッキに戻せず閲覧のみ」旨を表示） |
| 復帰 | **デッキへは戻せない**（アルバムから **💎5 削除** で px・かけら返還のみ） |

#### 6.3.4 ロスト時の選択肢（プロトタイプ UI）

| 状況 | 選択肢 |
|------|--------|
| 復活上限 **未到達** | **復活** / **思い出アルバムに保存** / **削除** |
| 復活上限 **到達** | **思い出アルバムに保存** / **削除**（「復活上限に達しました」表示） |
| N ロスト | 上記と同様 |

#### 6.3.5 active カードのデッキから外す操作

| 操作 | 挙動 |
|------|------|

## 7. 護符

**実装段階: `v1`**

### 7.1 概要

- カードに装備可能な消費アイテム。
- そのカードが **ロスト対象に選ばれたとき**、自動発動し **1回だけロストを免れる**。

### 7.2 ルール

| 項目 | 内容 |
|------|------|

| 発動 | ロスト対象選定時 **自動** → 護符消滅、カードは **active のまま** |
| 対人戦 | **有効** |

### 9.3 限界突破（★）

**実装段階: `v1`（プロトタイプ）**

| 項目 | 内容 |
|------|------|
| 手段 | **かけら合計**（専用＋汎用、**1個=1価値**）。★アップ時はかけらのみ。**N=10 / R=15 / SR・UR=20** |
| レア昇格（4回目相当） | 上記かけらに加え **💎 追加**（`LIMIT_BREAK_RARITY_JEWEL_COST`: N→R **50** / R→SR **100** / SR→UR相当 **200**） |
| UI | 詳細オーバーレイで **専用・汎用の使用数を個別選択**（合計は現在レア度に応じる）。表示は [§16.9](#169-限界突破ui) |
| 同一レア内 | ★1 → ★2 → ★3（最大 **3回**） |
| 4回目相当 | **レア1段階アップ** ＋ BP 増 ＋ ★リセット（★0） |
| 初版上限 | **UR 実装まで SR★3 が上限**（4回目昇格は不可） |
| 表示 | カードに **★1〜3** を表示 |

**資源の種類**

| 種類 | 入手 | 使用対象 |
|------|------|----------|
| **属性かけら** | 勝利時の **戦利品選択**（§4.1）。選択カードの属性に +1 | **同一属性**のカードのみ |
| **汎用かけら** | レベル **L ≡ 4 (mod 5)** かつ **L ≥ 5**（9, 14, 19, 24, 29, 34, 39, 44, 49 …） | **全属性**のカード（属性かけらの代用） |

- **Lv10 では汎用かけらを配布しない**（デッキ2解放に枠を使用。§10.4）。
- **Lv20, 30, 40, 50** では汎用かけらの代わりに **護符 ×1** を配布（§10.4）。
- 限界突破1回につき **現在レア度に応じたかけら数**（`getLimitBreakShardsRequired`）。専用・汎用は **1:1** で合算。
- 実行時はプレイヤーが選択した内訳で消費（ステッパーで調整。合計は常に10）。
- 戦利品かけらは **N=1, R=2, SR=3**（`GRAVEYARD_SHARD_REWARD`）。
- 将来ショップ: 属性かけらパック（💎）は **ショートカット** として検討可。

**戦略的意図**

- 育成したい属性のカードがある場合、相手墓地から **同属性の戦利品** を選ぶ動機が生まれる。
- px 最大化と属性かけらの **トレードオフ** が戦利品 UI に集約される。

**昇格例（UR 実装後）**

```
N★3 --(4回目)--> R★0
R★3 --(4回目)--> SR★0
SR★3 --(4回目)--> UR★0  （UR 実装後）
...
```

### 9.4 BP と表示

| イベント | BP |
|----------|-----|
| ★アップ・レア昇格 | **均等加算**（`LIMIT_BREAK_BP_GAIN_RATE` × 基礎BP、1回ごとに同じ増加量） |
| ~~降格復活~~ | ~~降格後レアの基礎 BP へ再計算~~（**廃止** — 2026-06-20） |

- 基礎 BP は `getCardFoundationBp`（色バランス + hash、レア・★適用前）。限界突破のたびに `round(基礎BP × 0.03)` を加算（最低1）。
- N★0→SR★3 は **11回** の限界突破。基礎BP=100 なら **1回 +3 BP** で最終 **133**。
- レア枠・ラベル・★表示は **オーバーレイ**（ピクセル本体は原則変更しない）。

## 13. 対人戦（将来）

> **ステータス**: 本章は **将来構想**。数値・フロー・データ出所は **未確定** であり、実装前に詳細を詰める。  
> **優先順位**: リアルタイムオンライン対人の前に、**オフライン対人（ゴーストデッキ）** を想定する（§13.2）。

### 13.1 対人戦の共通ルール（方針）

CPU戦と **経済ルールは同型** とする（敗北ペナルティ・戦利品・護符・初心者保護など）。

| 項目 | 内容 |
|------|------|
| 勝利報酬 | 相手墓地から **自分が1枚選択** → **px ＋ 属性かけら**（§4.1） |
| 敗北 | 自軍墓地から **相手が1枚選択** → Lost（§4.2） |
| 護符 | 有効 |
| 演出 | 相手の選択が可視（§5.3） |

- オフライン対人では「相手」は **ゴースト（他ユーザーのデッキスナップショット）** だが、敗北時の墓地選択は **相手側の意図を再現する** 形を目指す（AI による模倣か、記録済み選択の再生かは **TBD**）。
- 履歴再戦（§4.1.2）とは別モード。履歴再戦は戦績非記録・ロストなし・生存 px のみ。

### 13.2 オフライン対人戦（ゴーストデッキ）— 将来構想

#### 背景・目的

| 課題 | ゴーストデッキ案 |
|------|------------------|
| リリース直後は同時接続ユーザーが少ない | **非同期** に他ユーザーのデッキをプールし、自分のタイミングで対戦 |
| リアルタイムマッチは待ち時間・人口依存 | CPU戦と **同じ画面フロー**（デッキ選択 → 配置 → バトル）で遊べる |
| 実在デッキはレア・限界突破で強くなりうる | **レベル・戦力マッチ** ＋ 必要なら **BP 再計算** で体感の公平性を補正 |

UI 上は `BattleHubScreen` の **「対人戦（オフライン）」**（現状 disabled・準備中）がこのモードの入口を想定する。

#### 基本フロー（案）

1. プレイヤーがバトルデッキを選択（CPU戦と同様）。
2. サーバー／ローカルプールから **ゴーストデッキ** を1件選出（マッチング条件は下記）。
3. 必要に応じて `rescaleDeckBp` で相手デッキの **BP のみ** を補正。
4. 以降は CPU戦と同型のバトルフロー。勝敗・戦利品・Lost は §13.1。

#### マッチング（3段階・案）

| 段階 | 内容 | 備考 |
|------|------|------|
| **1. 候補検索** | **ユーザーレベル** と **デッキ戦力**（`computeDeckPower`）が近いデッキを優先 | CPU戦の `buildBalancedCpuDeck`（プレイヤー戦力に合わせた生成）と **同思想を実在デッキ版に置き換える** イメージ |
| **2. 戦力差の補正** | それでも差がある場合、**低い側のレベル**（または挑戦側のレベル — **TBD**）で `rescaleDeckBp` し、バトル上の BP を揃える | 履歴再戦の `prepareHistoryOpponentDeck` と同系（`historyRematch.ts`） |
| **3. 候補不足** | リリース直後などプールが薄いときは、戦力差マッチも許容し **2 で補正**。それでも無い場合は **CPU 生成デッキへフォールバック**（**TBD**） | 人口が増えたら 1 の厳しさを上げる想定 |

#### BP 補正で揃うもの / 揃わないもの（案）

| 補正で揃う（バトル公平） | 補正では揃わない（経済・演出） |
|--------------------------|--------------------------------|
| 戦闘中の **BP** | **ピクセル絵・塗り数・色数**（戦利品 px はここから算出） |
| レベル帯での体感強さ | **レア・★**（かけら報酬 N=1/R=2/SR=3） |
| | 限界突破 **回数**（BP 加算は再計算に含まれるが、デッキの「育成度」は残る） |

BP 補正は **一方的な押し負けを避ける** ためのもの。相手の丁寧な絵による **戦利品 px の増加** は対人らしさとして残す想定（§10.4 の経済設計意図と整合）。

#### 経済への影響（構想レベル）

| 側面 | 想定 |
|------|------|
| 収入 | ユーザーデッキは満塗り・多色が多く、勝利時の戦利品 px が **CPU 模様より増えやすい** |
| 支出 | 互角マッチになりやすく **敗北・Lost が増えやすい**。相手による墓地選択で護符・復活の価値が上がる |
| px 創作コスト | CPU 中心の序盤は余りやすい帯。ゴースト対人が増えると収入↑と復活↑が同時に起き、**300/300/200 の px 設計が「ちょうどよい」帯に寄りやすい**（§10.4） |

数値バランスの再試算は **ゴースト対人の参戦率が見えてから** 行う。

#### 技術的な足がかり（現行コード）

| 要素 | 役割 |
|------|------|
| `rescaleDeckBp` / `recalculateCardBp` | 絵・属性・レア・★・限界突破回数は維持し、**BP のみ** ユーザーレベルに合わせ直す |
| `prepareHistoryOpponentDeck` | 履歴再戦で上記を適用済み。ゴースト対人の **BP 補正プロトタイプ** として参照可 |
| `computeDeckPower` | マッチングキーの候補（属性係数＋BP 合計） |
| `buildBalancedCpuDeck` | 戦力帯合わせの参照実装 |

#### 未確定事項（今後詰める）

| # | 論点 |
|---|------|
| G1 | **デッキの出所** — いつ・どの条件でプールに載せるか（公開デッキ、バトル履歴、オプトイン等）。プライバシーと鮮度 |
| G2 | **マッチングキー** — `userLevel` + `computeDeckPower` のみか、レア構成・属性分布も見るか |
| G3 | **BP 補正の基準レベル** — 常に挑戦側のレベルか、弱い方に合わせるか（履歴再戦は挑戦側＝現在プレイヤーレベル） |
| G4 | **敗北時の相手選択** — CPU ランダムではなく相手意図の再現方法（AI・記録再生等） |
| G5 | **戦利品のカードデータ** — 勝利時は墓地から選ぶのみで自デッキに保存しない（現行 §4.1）をオフライン対人でも踏襲するか |
| G6 | **戦績・履歴** — ゴースト対人を `battleHistory` に載せるか、別ログか |
| G7 | **広告** — 通常 CPU 戦と同じ §11 ルール（3回に1回）か、別枠か |
| G8 | **CPU 行動** — ゴースト側のターン操作は現行 CPU AI のままか、デッキ作者の行動傾向を反映するか（長期） |

### 18.1 プロトタイプ簡略実装（現行コード）

CPU 戦プロトタイプでは **`src/config/economy.ts`** の式を使用する。**復活ポーションは採用しない**（§6.3）。

#### 復活 px（`calcReviveCost`）

```
floor( 塗りマス数 × REVIVE_PAINTED_MULTIPLIER × レア倍率 × ★倍率 )
```

| 定数 / 倍率 | 値 |
|-------------|-----|
| `REVIVE_PAINTED_MULTIPLIER` | **3**（例: 塗り256 → 基礎768） |
| `REVIVE_CAP` | **3**（`reviveCount` が上限に達すると px 復活不可） |
| レア倍率 | `LOST_WEIGHT_RARITY` — N=1.0, R=1.3, SR=1.6（UR/L は SR 扱い） |
| ★倍率 | `LOST_WEIGHT_STARS` — ★0=1.0, ★1=1.15, ★2=1.3, ★3=1.5 |
| 塗り0 | 最低 **1px** |
| 丸め | **切り捨て**（`Math.floor`） |

| 操作 | 傾斜の基準 |
|------|------------|
| **復活** | **現レア**・**現★** |

~~**降格復活**~~ — **廃止**（2026-06-20）

#### 削除（`calcLostCardDeleteRewards`）

| 項目 | 内容 |
|------|------|
| 💎 コスト | `JEWEL_COST_DELETE = **5**` |
| px 返還 | `calcGraveyardPixelReward`（`floor(√塗り × 色係数)`）— 勝利戦利品と同式 |
| かけら返還 | `GRAVEYARD_SHARD_REWARD` — N=1, R=2, SR=3（カード属性へ付与） |

#### リネーム（編集保存時）

| 回数 | コスト |
|------|--------|
