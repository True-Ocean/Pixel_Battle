## 1. 目的とスコープ

### 1.1 目的

- 企画「簡単！真剣！お絵描きピクセルバトル！」の**コア体験**を、低コストなウェブプロトタイプで検証する。
- **5枚制限・前衛2/後衛3の配置・ターン戦闘・自作ドット絵・カード名**の感触を確認する。
- 戦闘ロジックを **UI と分離** し、将来の **Unity 移植** および **サーバー検証** に再利用しやすくする。

### 1.2 プロトタイプで作るもの

| 区分 | 内容 |
|------|------|
| プラットフォーム | ブラウザ（React + Vite + TypeScript） |
| データ保存 | `localStorage`（サーバーなし） |
| 対戦 | CPU のみ |
| 対戦 | CPU のみ |
| カード属性 | **戦闘実装済み**: 剣・盾・力・弓・両・毒・癒・氷・嵐・忍（全10属性）。定義は [ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)。**ユーザーレベル**に応じて色・キャンバス・ツール・属性が解放（[§5.9](#59-ユーザーレベル解放5nm-統一)） |
| ユーザープロフィール | 初回セットアップでユーザー名登録。Lv.1 から開始。ヘッダーに Lv + プログレスバー。バトル報酬で EXP 蓄積・レベルアップ（上限 **Lv50**） |

### 1.3 プロトタイプで作らないもの

- ★（スター）・**AI 属性決定**（[§5.4](#54-属性決定) 将来方針・未実装）・創作ボーナスの拡張（パレット・キャンバス解放は実装済み）
- メール・パスワード認証（将来 API / DB で実装。型・仕様のみ予約可）
- 全10属性の戦闘実装は完了（仕様は [ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)）
- AI による漢字1文字属性決定（[§5.4.2](#542-将来方針ai-による漢字1文字属性未実装)）
- 本番ロスト（カードデータの永久削除）
- 課金・殿堂入り・オンライン非同期対戦
- 画像ファイルのサーバー保存

## 2. 技術スタックと環境

### 2.1 採用技術

| 層 | 技術 |
|----|------|
| フロントエンド | React 18+ / Vite / TypeScript |
| 状態・UI | React（コンポーネント + state） |
| 永続化 | `localStorage` |
| パッケージ管理 | npm |

### 2.2 開発環境

- **Node.js**: インストール済み想定（確認時 v24.14.1）
- **エディタ**: Cursor / VS Code
- **初回セットアップ（実装時）**:
  ```bash
  npm create vite@latest . -- --template react-ts
  npm install
  npm run dev
  ```
- **スマホ実機確認**: 開発サーバーは `host: true` で LAN 上の端末からアクセス可能（例: `http://192.168.x.x:5173`）
- **PWA**: `vite-plugin-pwa` によりホーム画面追加に対応。アイコン生成は `npm run generate:pwa-icons`

### 2.3 推奨ディレクトリ構成

```
src/
  components/     # 画面（キャンバス、デッキ、戦闘UI、ConfirmDialog、SetupScreen）
  config/         # balance.ts / palette.ts（定数・パレット配置）
  game/           # 純粋な戦闘ロジック（UI非依存）
  canvas/         # 16×16 描画・flood fill・チェッカーボード
  card/           # 属性・BP 算出（色バランス + hash）、create / update
  storage/        # localStorage 読み書き
  user/           # ユーザープロフィール・EXP 加算（UI非依存）
  types/          # Card, Attribute, BattleState, UserProfile 等
```

**原則**: `game/` と `card/` は React に依存しない関数・型のみとし、Unity（C#）移植時は仕様書 + テストケースを参照して再実装する。

## 4. 機能一覧（プロトタイプ）

### 4.1 ユーザーフロー

0. **初回セットアップ**（未登録時のみ）：ユーザー名入力 → Lv.1 から開始
1. **マイデッキ**画面（最大5枚表示・並べ替え・削除・タップで編集）
2. **イメージ作成** / **カード編集**：解放済みキャンバス・パレット・ツールで描画。新規は **カード名入力** → 生成。編集は **イメージのみ** 変更 → 保存（[§5.8](#58-新規作成と編集)）
3. 戦闘開始：5枚を **前衛2枚・後衛3枚** に配置 → バトル開始
4. ターン制戦闘（CPU 対戦）
5. 勝敗 → EXP 加算（勝敗に関係なく **現在レベル + 倒した敵枚数**）→ 仮ロスト演出（該当時）→ 次戦闘前にデッキ自動復帰

### 4.2 画面要件（最低限）

- **初回セットアップ**：ユーザー名入力（1〜16文字）+ **はじめる**。メール・パスワードは表示しない
- ヘッダーに **ユーザー名 · Lv.N · プログレスバー**（EXP 数値は非表示。セットアップ完了後）
- **マイデッキ**一覧（5枚上限、新規作成・編集・削除・並べ替え）
- ドット描画エディタ（パレット・ツール）+ 新規時は名前入力 + 生成（属性・BP は自動算出）。編集はイメージのみ・BP は保存時レベルで再算出
- 戦闘前：5枚配置 UI（前衛2・後衛3）+ 30秒準備タイマー
- 5枚通しの盤面、ターンごとの行動カード選択、対象選択、**段階的な攻撃演出**（ダメージ表示 → BPカウントダウン → 次の攻撃）
- 中央ガイダンス帯（1行表示）、陣営別勝敗表示（WIN / LOSE）、終了後フッター操作
- プロトタイプ注記：「カードは保存されています（仮ロスト）」

### 4.3 マイデッキ画面 UI

| 項目 | 内容 |
|------|------|
| 画面タイトル | **マイデッキ** |
| サブ情報 | `最大5枚 · N / 5`。仮ロスト中は「仮ロスト演出中（データは保存済み）」を表示 |
| 通常時の操作 | カード行をタップ → **カード編集**画面へ。サブに「タップで編集」を表示 |
| 編集モード | ヘッダ右の **編集** / **完了** トグル。サブに「並べ替え・削除」を表示 |
| 並べ替え | ドラッグ＆ドロップ、または各行の **↑** / **↓** |
| 削除 | 行左の **−** → 確認ダイアログ（「戦績も失われます」）。確定で `localStorage` から削除 |
| 編集中の制限 | カードタップ編集・**新規カード作成**・**バトル** を無効化 |
| フッター操作 | **新規カード作成**（5枚未満）、**バトル**（5枚揃ったときのみ有効） |
| 5枚未満 | 「戦闘にはあと N 枚必要です（合計 5 枚）」を表示 |

### 4.4 イメージ作成 / カード編集画面 UI

| 項目 | 新規（イメージ作成） | 編集（カード編集） |
|------|----------------------|---------------------|
| 画面タイトル | **イメージ作成** | **カード編集** |
| 確定ボタン | **カード作成**（確認ダイアログ後に保存） | **保存**（即時） |
| 戻る | **マイデッキに戻る**（未保存内容は破棄） | 同左 |
| イメージ（`pixels`） | 編集可 | 編集可（**唯一の編集対象**） |
| カード名 | 入力可 | **表示のみ**（変更不可） |
| 属性（漢字1文字） | 作成時に自動決定 | **表示のみ**（変更不可） |
| キャンバスサイズ | 解放済みから選択 | **作成時固定**（変更不可） |
| 使用可能な色 | 現在のユーザーレベルで解放済みの色 | 同左（レベルアップ後に新色で仕上げ可能） |

**レイアウト（上から）**

1. 画面タイトル（この画面ではアプリ共通ナビ **非表示**）
2. **キャンバスサイズ選択** + **ミニプレビュー**（描画エリア上）
3. **描画ツール**（縦・左）+ **ドットキャンバス**（右）
4. **カラーパレット**（キャンバス下・横一列）
5. **カード名**（新規は入力欄、編集は読み取り専用）+ **属性表示**（編集時）+ 説明文
6. フッター固定：**カード作成** / **保存**、**マイデッキに戻る**

**説明文（カード名の下）**

- **新規作成**:
  > あなたが描いたイメージとカード名から、カードのステータスが自動生成されるよ
- **編集**:
  > イメージのみ変更できます。名前と属性を変えるには新しいカードを作成してください。保存すると、現在のレベルに応じて BP が再計算されます。

**カラーパレット（キャンバス下・8列）**

- 色スロット（index 0〜7、白〜桃）。未解放は **ロック枠** 表示
- 解放スケジュールは [§5.6](#56-パレットとレベル解放) 参照

**描画ツール（キャンバス左・縦並び）**

- `IDEAL_TOOL_ORDER`（`src/config/editorTools.ts`）の理想順で、**解放済みかつ実装済み** のみ上から表示
- Lv1: ペン・消しゴム・塗りつぶし・クリア / Lv7: 元に戻す / Lv12: やり直す / Lv17: 直線 / Lv22: 選択 / Lv27: 矩形 / Lv32: 円
- **スポイト**はコード実装済みだが **UI 非表示**（パレット色が大幅増加した将来に解放予定）

| ツール | 挙動 |
|--------|------|
| ペン | 選択色でマスを塗る（Lv1 解放。ドラッグで連続） |
| 色スウォッチ | **ブラシ色** を選択（ツールは切り替えない） |
| 消しゴム | タップしたマスを `null`（透明）に |
| 塗りつぶし | 同色相連領域を **flood fill** |
| クリア | 全マスを空に（チェッカーボード背景 + 「クリア」ラベル） |
| 元に戻す | 直前の編集状態に戻す（Lv7 解放） |
| やり直す | 元に戻した操作を取り消す（Lv12 解放。新規編集で future はクリア） |
| 直線 | ドラッグで2点間に直線（Lv17 解放。選択色で描画） |
| 選択 | 矩形範囲をドラッグで指定 → **フロート表示** → ドラッグで複製先を指定して確定（**元の絵は残る**。Lv22 解放） |
| 矩形 | ドラッグで矩形の外枠（Lv27 解放） |
| 円 | ドラッグで楕円の外枠（Lv32 解放） |
| スポイト | （将来）タップしたマスの色を取得。**現 UI 非表示** |

**キャンバスサイズ選択**

- **ドロップダウン** で **16 〜 解放済み上限** の整数から選択（未解放サイズは非表示またはロック）
- 新規作成のデフォルトは **現在レベルで使える最大サイズ**
- 既存カード編集時は **作成時の `canvasSize` 固定**（変更不可）

**キャンバス**

- 選択サイズの正方形グリッド（初期 16×16、最大 Lv50 で 36×36）。未塗りマスは **チェッカーボード**
- 画面上のキャンバス枠は CSS でほぼ固定。解像度が上がると 1 マスは小さくなる
- **カード表示**（デッキ・バトル等）は外枠サイズ固定。`canvasSize` に応じてセルがスケール（`image-rendering: pixelated`）
- サイズ変更時は既存ピクセルを **左上基準** で新グリッドにコピー
- ペン / 消しゴムはドラッグで連続操作（塗りつぶしは1タップ）
- 直線・矩形・円はドラッグでプレビュー → 離して確定（選択色）
- 選択は範囲ドラッグでフロート化 → ドラッグで複製先を指定して確定（元位置は保持。他ツールへ切替時はその位置で複製確定）

**バリデーション（確定前）**

| 条件 | メッセージ |
|------|------------|
| カード名が空 | カード名を入力してください |
| 塗りマス 0 | 1マス以上塗ってください |
| デッキ上限（新規のみ） | デッキは最大 5 枚です |

**新規作成の確認ダイアログ**

- タイトル: 「カードを作成しますか？」
- 本文: 「このイメージと名前でカードを作成します。よろしいですか？」
- 確定: **作成する**（primary スタイル） / キャンセル

**編集時のデータ保持**

- `updateCardFromDrawing`：`id`・`name`・`attribute`・`rarity`・`stars`・`wins`・`losses`・`reviveCount`・`createdAt`・`canvasSize` は維持
- `pixels` のみ更新可能。`bp` は **保存時のユーザーレベル** と絵内容（`bpBlend`）に応じて再算出
- 詳細は [§5.8 新規作成と編集](#58-新規作成と編集) を参照

## 5. カード作成仕様

### 5.1 入力

| 項目 | 必須 | 説明 |
|------|------|------|
| `name` | ○（新規のみ入力） | プレイヤーが付けるカード名（最大32文字）。**編集時は変更不可** |
| `pixels` | ○ | `canvasSize` の正方形グリッド。色は **現在レベルで解放済みのパレット色** のみ有効。未塗りは `null` |
| `canvasSize` | ○（新規のみ選択） | 解放済みサイズから選択。**編集時は作成時の値を維持** |

### 5.2 色比率の算出

塗られたマスのみを対象とする（未塗りは分母に含めない。実装時にコメントで固定すること）。

```
R = 赤マス数 / 塗ったマス合計
W = 白マス数 / 塗ったマス合計
B = 黒マス数 / 塗ったマス合計
（R + W + B = 1）
```

### 5.3 シード（hash）

- **入力文字列**: `name` + 正規化した `pixels`（JSON 文字列など、実装で1形式に固定）
- **処理**:  cryptographic hash または十分な衝突耐性の hash → 数値化（例: 0〜1 の `hashAttack`, `hashDefense`）
- **性質（プロトタイプ）**: 同じ名前 + 同じ絵 → **常に同じ** 属性・BP（決定論）
- **性質（将来・AI）**: 作成時に1回だけ判定し、結果を `Card.attribute` に保存。以降は編集しても属性は変わらない

### 5.4 属性決定

#### 5.4.1 プロトタイプ現行（Lv0: 攻撃 | 防御）

**方針**: 最多色ではなく **3色バランスと hash を常に合成**（Q14-B）。実装: `src/card/createCard.ts` の `deriveAttribute`。

```
色攻撃 = R + B × BLACK_ATTACK_WEIGHT   // 初期 0.7（黒は攻撃寄り）
色防御 = W + B × (1 - BLACK_ATTACK_WEIGHT)  // 初期 0.3

最終攻撃 = 色攻撃 × COLOR_WEIGHT + hashAttack × HASH_WEIGHT
最終防御 = 色防御 × COLOR_WEIGHT + hashDefense × HASH_WEIGHT

// 初期: COLOR_WEIGHT = 0.7, HASH_WEIGHT = 0.3（定数1か所、後から調整可）

attribute = 最終攻撃 >= 最終防御 ? "attack" : "defense"
```

- **新規作成時のみ** 属性を決定する。編集時は属性を変更しない（[§5.8](#58-新規作成と編集)）。

#### 5.4.2 将来方針（AI による漢字1文字属性）※未実装

カード作成ボタン確定時（**新規作成のみ**）に、以下を入力として AI（またはサーバー経由の同等ロジック）へ渡し、**漢字1文字**の属性を返させる。表示は `ATTRIBUTE_META`（`src/config/attributes.ts`）の `label` に対応させる。

**入力（コスト最小化を想定）**

| 項目 | 内容 |
|------|------|
| カード名 | プレイヤー入力 |
| イメージの見た目 | ASCII ピクセル表現などコンパクトな形式（将来 Vision API 併用も可） |
| 色数 | 使用した色の種類数 |
| カラーバランス | 各色の比率・色調の雰囲気 |
| 塗りピクセル数・密度 | 塗ったマス数、`canvasSize` に対する密度 |
| キャンバスサイズ | `canvasSize` |
| 候補属性 | **作成時のユーザーレベルで解放済み**の属性のみ（漢字リスト） |

**方針**

- 色合いのバランス・イメージの意味・カード名を総合して属性を決める（色だけでは決めない）
- 結果はカードに保存し、**編集では再判定しない**
- API 失敗時はプロトタイプと同様の hash + 色比率フォールバックを検討
- **色のレベル解放と属性の追加**: 色を Lv N で解放し、属性を Lv N+1 で追加する案を検討中（詳細は実装時に確定）

### 5.5 BP 決定

- **方針**: `hash` + **密度**（塗ったマス / グリッド総マス数）で、属性ごとの **BP レンジ** 内の値を決定。
- 密度が高いほどレンジの **上限側** に寄せる（線形補間。`src/config/balance.ts` の `computeCardBaseBp`）。
- **レンジの中心**は **ユーザーレベル** と属性でスケール（`getUserBaseBp`）。

| タイミング | BP の算出 |
|------------|-----------|
| **新規作成** | 作成時の `userLevel` でレンジを決定。`bpBlend`（hash + 密度）でレンジ内の値を決める |
| **編集保存** | **保存時の `userLevel`** でレンジを再スケール。`attribute`・`rarity` は固定。`bpBlend` は絵の変更で変わりうる |

- レベルアップ後に既存カードを編集保存すると、同じ絵でも BP レンジが伸びる。絵を直すと `bpBlend` も変化しうる。
- プロトタイプ用の具体レンジ（例、要プレイテスト）:

| 属性 | BP レンジ（案） |
|------|-----------------|
| attack | 70 〜 100 |
| defense | 55 〜 85 |

※上記数値は指示書時点の仮値。`src/config/balance.ts` で一元管理すること。

`PALETTE_16` の先頭10色がプレイ用。index 10〜15 は将来用に予約。

| 1 | 黒 | 1 |
| 2 | 赤 | 1 |
| 3 | 青 | **5** |
| 4 | 黄 | **15** |

|------|------------|-------------------|

**UI 配置**

- キャンバス下: 色 index 0〜7（8 枠固定、未解放はロック）

### 5.8 新規作成と編集

新規作成と編集は、プレイヤーにとって別の目的を持つ。

| | **新規作成** | **編集** |
|--|--------------|----------|
| 目的 | 名前・属性・サイズを含め **ゼロから** 新しいカードを得る | 同じ名前・属性のカードの **イメージを仕上げる** |
| イメージ | 自由に描く | 修正可（**唯一の編集対象**） |
| カード名 | 入力して決定 | **変更不可** |
| 属性 | 作成時に決定（現行: hash + 色比率 / 将来: AI） | **変更不可** |
| キャンバスサイズ | 解放済みから選択 | **作成時固定** |
| 色 | 現在レベルの解放色 | 現在レベルの解放色（レベルアップ後に新色で彩りを足せる） |
| BP | 作成時 `userLevel` | **保存時 `userLevel`** で再計算 |
| レア | 作成時抽選 | **維持**（編集では再抽選しない） |
| 戦績 | 0 で開始 | **維持** |

**プレイヤー動機（設計意図）**

- **編集**: レベルアップで増えた色で印象を少し変え、同じカードを育てる。名前・属性は固定のため、別キャラへの作り替えより **仕上げ** 向き。BP はレベルアップ後の保存で伸びる。
- **新規作成**: 別の名前・別の属性（将来は AI 判定）・より大きいキャンバスなど、編集では得られない要素を狙う。**新しい属性が欲しい場合は新規作成のみ**。

**実装**

- 新規: `createCardFromDrawing`
- 編集: `updateCardFromDrawing`（`src/card/createCard.ts`）
- 編集 UI での名前読み取り専用・属性表示は [§4.4](#44-イメージ作成--カード編集画面-ui) に従う（UI 反映は段階的に可）

## 6. デッキと永続化

### 6.1 デッキルール

- 保有上限: **5枚**
- ストックなし（常に5枚デッキのみ）

### 6.2 保存形式（Q18-C）

**永続化する（`localStorage`）**

```typescript
interface Card {
  id: string;              // uuid 等
  name: string;
  pixels: (string | null)[][];  // canvasSize×canvasSize、例: "#FF0000" | null
  canvasSize: number;           // 16〜上限の整数（未設定セーブは 16）
  attribute: "attack" | "defense";
  bp: number;              // 戦闘時の最大BPとして使用
  wins: number;
  losses: number;
  reviveCount: number;     // プロトタイプでは 0 固定で可
  createdAt: string;       // ISO 8601
}

interface SaveData {
  user: UserProfile | null;  // null = 初回セットアップ未完了
  deck: Card[];              // 最大5件
}

interface UserProfile {
  username: string;          // 1〜16文字（trim 後）
  level: number;             // 累計 exp から算出（表示用に保存）
  exp: number;               // 累計 EXP（UI では数値非表示）
}
```

**EXP・レベルアップ（`src/user/level.ts`）**

- 1 戦の獲得 EXP = **現在レベル + 倒した敵カード枚数**（0〜5。勝敗不問）+ **上振れボーナス**（勝利・相手デッキ戦力が高いときのみ、最大 +3）
- 上振れ: 相手戦力 / 自分戦力 が **≥1.1 → +1**, **≥1.2 → +2**, **≥1.3 → +3**（参考。ゲーム結果への直接影響は EXP のみ）
- レベル L → L+1 に必要な EXP = **`EXP_LEVEL_BASE + L`**（初期 `BASE=10` → Lv.1→2 は 11）
- 累計 EXP 到達点: `totalExp(L) = (L-1)*BASE + (L-1)*L/2`
- 上限 **`MAX_USER_LEVEL = 50`**。降格なし
- UI: プログレスバーのみ（`expInLevel / expToNextLevel`）。レベルアップ時バー長は不変でレンジ切替

**デッキ戦力（`src/card/power.ts`・参考表示のみ）**

- カード戦力 = `round(BP × bpWeight[属性] + flatBonus[属性])`
- デッキ戦力 = 5枚のカード戦力合計（編成補正なし）
- Lv0: 攻撃 `{ bpWeight: 1, flatBonus: 0 }` / 防御 `{ bpWeight: 1, flatBonus: 20 }`（盾能力の反映）
- マイデッキ画面サブ情報に **戦力 N** を表示。バトル結果・ダメージ計算には使わない

**将来（本番・サーバー側・プロトタイプでは未保存）**

```typescript
interface UserAuth {
  email: string;
  // password はクライアントに保持しない
}
```

**永続化しない（戦闘セッション内メモリのみ）**

- 現在 BP
- 盾の付与状態・使用済みフラグ
- 麻痺・毒・凍結・ステルス等 — [ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md) 参照。戦闘セッション内メモリのみ

### 6.3 ピクセル保存の段階

| 段階 | 形式 |
|------|------|
| プロトタイプ | JSON 2次元配列 |
| 本番 / DB | RLE 等へ変換（同じ論理データ） |

### 6.4 デッキ操作（マイデッキ）

| 操作 | 仕様 |
|------|------|
| 追加 | **新規カード作成** → イメージ作成画面。上限 5 枚 |
| 編集 | カード行タップ → カード編集画面。**イメージのみ**変更可。名前・属性は固定。保存で `pixels` と `bp` を更新（[§5.8](#58-新規作成と編集)） |
| 削除 | 編集モードの **−** → 確認後に削除（`wins` / `losses` も失う） |
| 並べ替え | 編集モードでドラッグまたは ↑↓。順序は `localStorage` に保存 |
| 表示順 | 並べ替え後の配列順がマイデッキ一覧・戦闘前手札の初期順に反映 |
| 戦力 | サブ情報 `戦力 N`（5枚合計。参考値） |

## 7. 戦闘前フロー

1. プレイヤーの **5枚** と CPU の **5枚** を表示する。
2. スマホ縦持ちを最優先し、上部に敵カード列（裏面5枚）、下部に自分カード列（表面5枚）を小さく横並び表示する。
3. 中央盤面に、敵・自分それぞれ **前衛2スロット / 後衛3スロット** を表示する。
4. CPU は準備開始時に5枚の配置をランダム決定する。上部の敵手札（裏面5枚）から、空の敵スロットへ **前衛左 → 前衛右 → 後衛左 → 後衛中央 → 後衛右** の順で1枚ずつ配置アニメーションする。
5. プレイヤーは下部カード列からタップ方式で自分の5枚を中央の自分スロットへ配置する。
6. 準備時間は **30秒**。準備画面表示と同時にカウント開始する。
7. 5スロット未配置の間、ボタンは **「ランダム配置」** と表示する。押すと未配置カードのみを空きスロットへランダム配置し、すぐには開始しない。
8. 5スロットすべて配置済みになると、ボタンは **「準備完了」** に変わる。押すとバトル開始。
9. 30秒経過時は、未配置カードを空きスロットへランダム配置し、そのままバトル開始する。
10. バトル開始時、CPUカードを **前衛左 → 前衛右 → 後衛左 → 後衛中央 → 後衛右** の順に1枚ずつ表面へ反転する。ガイダンスは **「カードオープン」** とする。

### 7.1 配置と操作

| 項目 | ルール |
|------|--------|
| 初期配置 | 前衛2枚、後衛3枚 |
| 自分の盤面向き | 前衛2枚が中央寄り、後衛3枚が手前 |
| 敵の盤面向き | 前衛2枚が中央寄り、後衛3枚が奥 |
| 操作方式 | ドラッグではなくタップ方式 |
| 選択解除 | 選択中の同じカードを再タップ |
| 未配置カード → 空きスロット | 配置 |
| 未配置カード → 埋まっているスロット | 入れ替え（元カードは未配置列へ戻る） |
| 配置済みカード → 空きスロット | 移動 |
| 配置済みカード → 埋まっているスロット | 入れ替え |
| 戻し操作 | 配置済みカードを未配置列へ戻す専用操作はなし |
| 時間切れランダム配置 | 既に配置済みのカード位置は動かさない |
| 手札表示 | 未配置カードは属性ラベル付きで表示。配置のたびに手札は中央揃え |
| カウントダウン | 中央ガイダンス帯に1行で表示（`バトル準備：…（残り N 秒）`） |
| スロット枠 | 配置中は点線枠を表示。双方5枚配置完了後、カードありスロットの枠は非表示 |
| 画面タイトル | 「バトル準備」「バトル！」の画面上部タイトルは表示しない |

## 8. 戦闘ルール

### 8.1 試合の流れ

- **1ターン** を繰り返す。
- **終了条件**: どちらか一方の **5枚すべてが BP 0**。
- **勝利**: 相手の5枚すべて撃破。
- **敗北**: 自分の5枚すべて撃破。
- 撃破されたカードは盤面スロットから外れ、墓地へ移動する。元スロットは空きスロットになる。

### 8.2 ターン開始：行動カード選択

| ルール | 内容 |
|--------|------|
| 選択枚数 | 1ターンに行動できるカードは各陣営 **1枚だけ**（**行動カード**） |
| プレイヤー / CPU | 行動可能なカードから1枚を選ぶ |
| 同時解決 | 双方の選択を揃えたうえで [ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#6-ターン処理順) の順に解決 |
| 剣・前衛 | 近接攻撃（敵前衛） |
| 盾・前衛 | 近接 **または** 盾付与（§9 / ATTRIBUTE_SPEC） |
| 凍結 | 凍結中のカードは **行動カードに選べない** |

属性ごとの行動可否・対象は **[ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)** を正とする。

### 8.3 攻撃対象（概要）

| 攻撃種別 | 対象 |
|----------|------|
| 近接攻撃 | 敵前衛（剣・盾・力・両・毒・氷・前衛の弓 等） |
| 弓 | 敵前衛・後衛（忍ステルス除く）。与ダメ currentBp **100%**・無反撃（ATTRIBUTE_SPEC §4.4） |
| 嵐 | 生存敵からランダム2体（別ユニット）。与ダメ currentBp **50%**・無反撃（ATTRIBUTE_SPEC §4.9） |

### 8.4 ターン処理：フェーズ

詳細・数値は **[ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#6-ターン処理順)** を参照。

```
【ターン開始 — 行動選択の前】
① ターン見出し・毒 DoT（§5.1）
② 撃破・前衛補充（DoT 起因。プレイヤーは補充必須）

【行動確定後】
③ 行動カード選択（各陣営1枚）※ UI 上は ①② の後
④ 癒（選ばれた側・先行・未実装）
⑤ 盾付与（選ばれた側）
⑥ 攻撃（弓・近接・両・嵐 等）
⑦ 撃破・墓地移動
⑧ 前衛補充（戦闘起因。弓前衛上がり時 BP 半減なし）
```

※ 詳細・番号対応は [ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#61-1ターン全体の流れ) を正とする。

### 8.5 前衛補充

配置を以下のように表す。

```
前衛: A B
後衛: C D E
```

| 空き前衛 | 上がれる後衛 |
|----------|--------------|
| A（左前衛） | C（左後衛）または D（中央後衛） |
| B（右前衛） | D（中央後衛）または E（右後衛） |

- D は左右どちらにも上がれる共有候補。
- 片側だけ空いている場合、対応候補がいなければ空きのまま次ターンへ進む。
- 両前衛が空いた場合も同じ対応ルールを守る。
- 複数候補がある場合、プレイヤーは補充順と移動先を自由に選べる。
- 候補が **1枚だけ** の場合は自動で前衛へ補充する（ユーザー操作不要）。
- 候補が **2枚** の場合のみ、ユーザーに選択を促す。
- CPU は対応候補からランダムに上げる。
- 補充操作はタップ方式: 上げる後衛カードをタップ → 同じカードを再タップで確定（1回目で選択、2回目で確定）。
- ガイダンス文言:
  - 未選択時: **「前衛に移動する後衛カードを選択」**
  - 選択後: **「もう一度タップで確定」**

### 8.6 撃破後

- BP 0 のカードはその試合では **もう選択できない**。
- 撃破カードは墓地に小さく表示し、グレーアウトまたは撃破ビジュアルを付ける。
- 自分と敵の墓地は分けて表示する。
- 盤面レイアウトは、準備中・バトル中・墓地移動後も固定する。

### 8.7 試合終了表示

| 項目 | 内容 |
|------|------|
| 中央ガイダンス | `TURN N：バトル終了！`（勝敗文言は出さない） |
| 敵陣営エリア | 墓地＋敵スロット全体の中央に **WIN** または **LOSE** を大きく重ね表示 |
| 味方陣営エリア | 味方スロット＋墓地全体の中央に **WIN** または **LOSE** を大きく重ね表示 |
| スタイル | グローは使わず、鮮やかな塗り＋太い縁取り（WIN＝黄、LOSE＝赤系） |
| 対人戦（将来） | 各クライアントで自分の陣営側に WIN/LOSE を表示。中央は双方共通の終了メッセージ |
| 終了後操作 | `デッキ` / `新規バトル` を画面下端に固定。盤面本体はスクロール可能 |

### 8.8 攻撃解決の詳細

#### 近接・反撃（剣/盾 Lv0 ＋ 全属性共通）

- **原則**: 攻撃ダメージは **currentBp** 基準（[ATTRIBUTE_SPEC §2.2](./ATTRIBUTE_SPEC.md#22-ダメージの基準)）。
- **相打ち**（同一2枚が互いに近接の行動カード）: 双方 **100%** の相互ダメージ。
- **一方的**: 攻撃側 **100%**、反撃 **50%**。
- 詳細・弓/両/嵐/毒/氷/忍は **[ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)**。

#### 盾の相互作用

| 状況 | 結果 |
|------|------|
| 防御側のみ盾あり | 防御側はダメージ0、盾消費。攻撃側は通常通り反撃ダメージ |
| 攻撃側のみ盾あり | 攻撃側は反撃ダメージ0、盾消費。防御側は通常通りダメージ |
| 双方とも盾あり | 双方の盾が消え、双方BPは変わらない |

#### 複数攻撃の処理順

同一ターンに複数の近接攻撃が発生する場合:

1. 互いに同じ2枚のカードが攻撃し合う場合は **1戦闘** としてまとめる（両矢印）。
2. 残りの戦闘は **1件ずつ順番** に処理する。
3. 戦闘の優先順位は **攻撃を仕掛けたカードのBPが高い順**。
4. 同率なら双方の最大BPが高い順 → さらに同率なら防御側BPが高い順。

UI 演出もこの処理順に従い、各戦闘ごとに「ダメージ表示 → BPカウントダウン」を行う。

#### 複数攻撃の処理順

1. 互いに同じ2枚が近接の行動カード → **1戦闘**（相打ち / `bidirectional`）。
2. 残りは **1件ずつ** 処理。
3. 優先: **行動カードの currentBp 高い順**（同率は [ATTRIBUTE_SPEC §7 tie-break](./ATTRIBUTE_SPEC.md#7-同率時の解決順tie-break)）。

UI 演出もこの順に従う。

## 10. CPU 仕様

### 10.1 配置

- CPU の5枚配置は、プロトタイプではランダム。
- 準備中は上部の敵手札に裏面5枚を表示し、空スロットへ1枚ずつ飛ばして配置する。
- 配置順は **前衛左 → 前衛右 → 後衛左 → 後衛中央 → 後衛右**。
- バトル開始時に同順で1枚ずつ表面へ反転する。

### 10.2 行動選択

- 初期実装では、CPUの通常ターン行動はランダムでよい。
- 実行可能な行動候補を列挙し、その中からランダムに1つ選ぶ。
- 攻撃行動の場合、敵前衛の生存カードからランダムに対象を選ぶ。
- 盾付与行動の場合、盾を持っていない生存味方からランダムに対象を選ぶ（**付与者自身は除く**）。
- 将来は、前衛保護・低BP保護・撃破可能攻撃などを評価するスコア式AIへ差し替えてよい。

### 10.3 前衛補充

- 空いた前衛に対して、対応候補からランダムに上げる。
- 候補がいなければ補充しない。

### 10.4 CPU デッキ

- プロトタイプ: 生成ルールまたは `src/data/cpu-decks.json` 等の **固定デッキ** で可。
- 将来: 他ユーザーの5枚メタデータを取得し、同じルールで CPU が操作。

### 12.6 画面・演出指針

#### レイアウト

- スマホ縦持ちを最優先レイアウトとする。
- PC では body 背景の上に **phone シェル** として中央配置する（`--phone-shell-w: 26.5rem`、バトル画面は `--phone-shell-battle-w: 22rem`）。
- 盤面レイアウトは準備中・バトル中・墓地移動後も固定する。
- 手札・配置スロット・バトル中・墓地の **すべてのカード** は、同一の内部レイアウト（パーツ位置・サイズ・マージン）で表示する。
- カードサイズは `--formation-card-w` / `--formation-card-h` で統一する。
- 中央はガイダンス表示と矢印・ダメージ演出のために使う。

#### イメージ作成 / カード編集画面

- 画面上部は共通ブランドタイトルのみ。**画面切替ナビは非表示**。
- パレット → キャンバス → カード名 → 説明文の間に **約 0.7〜0.85rem** の縦余白を確保する。
- フッター（**カード作成** / **保存**、**マイデッキに戻る**）は画面下部に固定配置する。

#### 戦闘準備・戦闘画面

- 戦闘準備・戦闘中は画面上部の「バトル準備」「バトル！」タイトルおよびアプリ共通ヘッダを **非表示** とする。
- 中央ガイダンスは **1行帯**（ピンク〜紫グラデーション、左右アクセントライン）。`TURN N：操作ヒント` 形式で常に1行。
- 準備フェーズは盤面・手札を `formation-setup-body` でスクロール可能にし、下部の `ランダム配置` / `準備完了` は固定。
- バトル終了後は盤面を `formation-battle-body` でスクロール可能にし、`デッキ` / `新規バトル` を下端固定。
- スロット枠は配置中のみ点線表示。双方配置完了後およびバトル中は、カードありスロットの枠を非表示（`is-frameless`）。
- 試合終了時、中央は `バトル終了！`、各陣営エリア（`formation-field-cpu` / `formation-field-player`）に WIN/LOSE を重ねる。

#### ガイダンス・操作

- ガイダンス文言は短い日本語で統一する。
- ターン表示は `TURN 1` のような英字表記とする。
- 行動カード選択中は選択カードをハイライト表示する。
- 選択中の同じカードを再タップすると選択解除する（専用の「選択解除」ボタンは置かない）。
- 防御属性・前衛で攻撃と盾付与の両方が可能なとき、**行動ボタンは出さない**。敵前衛タップで攻撃、盾なし味方タップで盾付与。
- 攻撃対象選択時は **敵前衛** を赤系でハイライトする。盾付与対象は **味方** を緑系でハイライトする。

#### 盾・バフ表示

- 盾状態はカード全体を覆う半透明表示ではなく、**カード左上の小さなバフアイコン**（🛡）で示す。
- 将来のバフ・デバフも同じ左上エリアに並べられる構造とする。
- BP は右上にライトな色のバッジ表示とする。

#### ターン演出（攻撃・盾）

1. **盾付与フェーズ**（該当時）  
   緑系の矢印で付与元→付与先を表示する。

2. **攻撃フェーズ**（1戦闘ずつ順番に）  
   各戦闘について次の順で再生する。

   | サブフェーズ | 内容 |
   |--------------|------|
   | ダメージ表示 | 関係するカードの中央に `−85` などのダメージ数値を大きく表示。盾で0の場合は非表示 |
   | BP変化 | ダメージ表示を消し、右上BPを **from→to** でカウントダウンアニメーション |
   | 次へ | 上記完了後、次の戦闘へ。全戦闘完了後にターン終了処理へ |

3. **矢印**
   - 攻撃は **カードの端から端** へ向ける（中央同士ではなく、矩形外周の交点を始点・終点とする）。
   - 相互攻撃の場合は両矢印（↔）表示。
   - 対象が異なる同時攻撃は片矢印2本を同時表示。
   - **BPカウントダウン中は攻撃矢印を非表示** にする（ダメージ表示フェーズでのみ表示）。
   - ダメージ表示は矢印レイヤーより上の専用レイヤーに描画し、矢印に隠れないようにする。

4. **その他**
   - 処理中のカードは揺れエフェクトを付ける。
   - バトルログは画面に出さない。内部ログ・構造化イベントはリプレイ用に維持する。

| `CLASH_MS.bp` | 520 | BPカウントダウン時間 |
| `CLASH_MS.exit` | 450 | ターン演出終了後の待機 |

#### resolveTurn 戻り値（攻撃演出用）

各 `AttackPlayback` には以下を含める。

- 攻撃側・防御側の位置（`fromSide` / `fromPosition` / `toSide` / `toPosition`）
- 防御側ダメージ・BP変化（`damage`, `bpFrom`, `bpTo`）
- 攻撃側ダメージ・BP変化（`attackerDamage`, `attackerBpFrom`, `attackerBpTo`）
- 相互攻撃フラグ（`bidirectional`）
- 処理後状態（`stateAfter`）
