## 1. 目的とスコープ

### 1.1 目的

- 企画「簡単！真剣！お絵描きピクセルバトル！」の**コア体験**を、低コストなウェブプロトタイプで検証する。
- **5枚制限・前衛2/後衛3の配置・ターン戦闘・自作ドット絵・カード名**の感触を確認する。
- 戦闘ロジックを **UI と分離** し、将来の **Unity 移植** および **サーバー検証** に再利用しやすくする。

### 1.2 プロトタイプで作るもの

| 区分 | 内容 |
|------|------|
| プラットフォーム | ブラウザ（React + Vite + TypeScript） |
| データ保存 | `localStorage`（サーバーなし） |
| 対戦 | CPU のみ |
| カード属性 | **戦闘実装済み**: 剣・盾・力・弓・両・毒・癒・氷・嵐・忍・照（全11属性）。定義は [ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)。**ユーザーレベル**に応じて色・キャンバス・ツール・属性が解放（[§5.9](#59-ユーザーレベル解放5nm-統一)） |
| オーディオ | **BGM** 実装済み（設定画面で ON/OFF）。**効果音（SE）** は [SFX_SPEC.md](./SFX_SPEC.md) に v1 仕様定義・**実装・素材は未着手** |
| ユーザープロフィール | 初回セットアップでユーザー名登録。Lv.1 から開始。ヘッダーに Lv + プログレスバー。バトル報酬で EXP 蓄積・レベルアップ（上限 **Lv50**） |

### 1.3 プロトタイプで作らないもの

- ★（スター）・**AI 属性決定**（[§5.4.2](#542-将来方針ai-による漢字1文字属性) 将来方針・未実装）・創作ボーナスの拡張（パレット・キャンバス解放は実装済み）
- メール・パスワード認証（将来 API / DB で実装。型・仕様のみ予約可）
- 全11属性の戦闘実装は完了（仕様は [ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)）
- AI による漢字1文字属性決定（[§5.4.2](#542-将来方針ai-による漢字1文字属性未実装)）
- 本番ロスト（カードデータの永久削除）
- 課金・殿堂入り・オンライン非同期対戦
- 画像ファイルのサーバー保存

## 2. 技術スタックと環境

### 2.1 採用技術

| 層 | 技術 |
|----|------|
| フロントエンド | React 18+ / Vite / TypeScript |
| 状態・UI | React（コンポーネント + state） |
| 永続化 | `localStorage` |
| パッケージ管理 | npm |

### 2.2 開発環境

- **Node.js**: インストール済み想定（確認時 v24.14.1）
- **エディタ**: Cursor / VS Code
- **初回セットアップ（実装時）**:
  ```bash
  npm create vite@latest . -- --template react-ts
  npm install
  npm run dev
  ```
- **スマホ実機確認**: 開発サーバーは `host: true` で LAN 上の端末からアクセス可能（例: `http://192.168.x.x:5173`）
- **PWA**: `vite-plugin-pwa` によりホーム画面追加に対応。アイコン生成は `npm run generate:pwa-icons`

### 2.3 推奨ディレクトリ構成

```
src/
  audio/          # BGM・SE プレイヤー（UI 層から利用。game/ 非依存）
  components/     # 画面（キャンバス、デッキ、戦闘UI、MissionScreen、ConfirmDialog、SetupScreen）
  config/         # balance.ts / missions.ts / palette.ts / bgm.ts / sfx.ts（定数）
  mission/        # ミッション進捗・受取・JST リセット（UI非依存）
  game/           # 純粋な戦闘ロジック（UI非依存）
  canvas/         # 16×16 描画・flood fill・チェッカーボード
  card/           # 属性抽選（rollAttribute）・BP 算出・create / update / attributeChange
  storage/        # localStorage 読み書き
  user/           # ユーザープロフィール・EXP 加算・missionState（UI非依存）
  types/          # Card, Attribute, BattleState, UserProfile 等
```

**原則**: `game/` と `card/` は React に依存しない関数・型のみとし、Unity（C#）移植時は仕様書 + テストケースを参照して再実装する。

## 3. 将来アーキテクチャ（参考）

本プロトタイプの決定事項は、以下の将来構成と矛盾しないこと。

| 段階 | 技術 | 役割 |
|------|------|------|
| プロトタイプ | React + Vite | ルール検証・バランス・UX |
| 本番クライアント | Unity | バトルアニメ・パーティクル・ドット演出 |
| サーバー | API + DB（例: Postgres） | ユーザーデータ、非同期対戦、戦績 |
| ストレージ方針 | ピクセルは RLE 等の文字列 + メタデータのみ | 画像ファイルはサーバーに保存しない |

## 4. 機能一覧（プロトタイプ）

### 4.1 ユーザーフロー

0. **初回セットアップ**（未登録時のみ）：ユーザー名入力 → Lv.1 から開始
1. **マイデッキ**画面（最大5枚表示・並べ替え・削除・タップで編集）
2. **イメージ作成** / **カード編集**：解放済みキャンバス・パレット・ツールで描画。新規は **カード名入力** → 生成（属性は **ランダム抽選**）。編集は **イメージ・名前・キャンバス拡大** → 保存（[§5.8](#58-新規作成と編集)）
3. **バトル**タブ：CPU戦 → デッキ選択 → 5枚を **前衛2枚・後衛3枚** に配置 → バトル開始
4. ターン制戦闘（CPU 対戦）
5. 勝敗 → EXP 加算（勝敗に関係なく **現在レベル + 倒した敵枚数**）→ ロスト抽選（該当時）→ 戦利品選択（勝利時）
6. **ミッション**タブ：デイリー / ウィークリー / 常設 / ビギナー（未完了時）の達成・報酬受取（[§4.8](#48-ミッション)）
7. **バトル**ハブ右上 **📊** から **戦績** → **バトル履歴** 再戦（[§11.4](#114-バトル履歴再戦)）

### 4.2 画面要件（最低限）

- **初回セットアップ**：ユーザー名入力（1〜10文字）+ **はじめる**。メール・パスワードは表示しない
- ヘッダーに **ユーザー名 · Lv.N · プログレスバー**（EXP 数値は非表示。セットアップ完了後）。**三本線メニュー**は水色プロフィールバー内右端
- **マイデッキ**一覧（5枚上限、新規作成・編集・削除・並べ替え）
- ドット描画エディタ（パレット・ツール）+ 新規時は名前入力 + 生成（属性・BP は自動算出。属性は **解放済みからランダム**）。編集はイメージ・名前・キャンバス拡大・BP は保存時再算出
- 戦闘前：5枚配置 UI（前衛2・後衛3）+ 30秒準備タイマー
- 5枚通しの盤面、ターンごとの行動カード選択、対象選択、**段階的な攻撃演出**（ダメージ表示 → BPカウントダウン → 次の攻撃）
- 中央ガイダンス帯（1行表示）、陣営別勝敗表示（WIN / LOSE）、終了後フッター操作
- 敗北時: 墓地から1枚 **Lost**（Lv5+。護符で免れ可）。復帰は [ECONOMY §6.3](./ECONOMY_SPEC.md#63-復帰オプション)

**レイアウト（上から）**

1. 画面タイトル（この画面ではアプリ共通ナビ **非表示**）
2. **キャンバスサイズ選択** + **ミニプレビュー**（描画エリア上）
3. **描画ツール**（縦・左）+ **ドットキャンバス**（右）
4. **カラーパレット**（キャンバス下・**2行×10列**）
5. **カード名**（新規は入力欄、編集は **常時編集可**）+ **属性表示**（編集時・読み取り専用）+ 説明文
6. フッター固定：**カード作成** / **保存**（必要時 **拡大** 🪙 / **リネーム** 🪙 内訳表示）、**マイデッキに戻る**
7. 編集保存後（BP 変化あり）: `EditorSaveBpConfirmModal` — BP 増減 + 課金時の保有コイン/ジュエル表示

**説明文（カード名の下）**

- **新規作成**:
  > あなたが描いたイメージとカード名から、カードのステータスが自動生成されるよ
- **編集**:
  > イメージ・カード名・キャンバスサイズ（拡大のみ）を変更できます。属性はカード詳細のリタッチ/セレクトで変更できます。保存すると、現在のレベルに応じて BP が再計算されます。

**カラーパレット（キャンバス下・2行×10列）**

- 計20スロット（index 0〜19）。**列ごとに用途が揃う**（白の下に灰、青の下に薄青 等）
- index 0〜7（白〜桃）: レベル解放。未解放は **ロック枠** ＋ 必要 Lv をツールチップ表示
- index 8〜19: ショップ追加色。未購入は **鍵アイコン**。Lv50 未満は「Lv50で購入可能」、Lv50+ はタップで **購入モーダル**（`PaletteUnlockModal`）
- 解放スケジュール・価格帯は [§5.6](#56-パレットとレベル解放) 参照

**描画ツール（キャンバス左・縦並び）**

- `IDEAL_TOOL_ORDER`（`src/config/editorTools.ts`）の理想順で、**解放済みかつ実装済み** のみ上から表示
- Lv1: ペン・消しゴム・塗りつぶし・クリア / Lv7: 元に戻す / Lv12: やり直す / Lv17: 直線 / Lv22: 選択 / Lv27: 矩形 / Lv32: 円
- **スポイト**はコード実装済みだが **UI 非表示**（パレット色が大幅増加した将来に解放予定）

| ツール | 挙動 |
|--------|------|
| ペン | 選択色でマスを塗る（Lv1 解放。ドラッグで連続） |
| 色スウォッチ | **ブラシ色** を選択（ツールは切り替えない） |
| 消しゴム | タップしたマスを `null`（透明）に |
| 塗りつぶし | 同色相連領域を **flood fill** |
| クリア | 全マスを空に（チェッカーボード背景 + 「クリア」ラベル） |
| 元に戻す | 直前の編集状態に戻す（Lv7 解放） |
| やり直す | 元に戻した操作を取り消す（Lv12 解放。新規編集で future はクリア） |
| 直線 | ドラッグで2点間に直線（Lv17 解放。選択色で描画） |
| 選択 | 矩形範囲をドラッグで指定 → **フロート表示** → ドラッグで複製先を指定して確定（**元の絵は残る**。Lv22 解放） |
| 矩形 | ドラッグで矩形の外枠（Lv27 解放） |
| 円 | ドラッグで楕円の外枠（Lv32 解放） |
| スポイト | （将来）タップしたマスの色を取得。**現 UI 非表示** |

**キャンバスサイズ選択**

- **ドロップダウン** で **16 〜 解放済み上限** の整数から選択（未解放サイズは非表示またはロック）
- 新規作成のデフォルトは **現在レベルで使える最大サイズ**
- 既存カード編集時は **作成時 `canvasSize` 以上** のみ選択可（[§5.8](#58-新規作成と編集)）

**キャンバス**

- 選択サイズの正方形グリッド（初期 16×16、最大 Lv50 で **34×34**）。未塗りマスは **チェッカーボード**
- 画面上のキャンバス枠は CSS でほぼ固定。解像度が上がると 1 マスは小さくなる
- **カード表示**（デッキ・バトル等）は外枠サイズ固定。`canvasSize` に応じてセルがスケール（`image-rendering: pixelated`）
- サイズ変更時は既存ピクセルを **左上基準** で新グリッドにコピー
- ペン / 消しゴムはドラッグで連続操作（塗りつぶしは1タップ）
- 直線・矩形・円はドラッグでプレビュー → 離して確定（選択色）
- 選択は範囲ドラッグでフロート化 → ドラッグで複製先を指定して確定（元位置は保持。他ツールへ切替時はその位置で複製確定）

**バリデーション（確定前）**

| 条件 | メッセージ |
|------|------------|
| カード名が空 | カード名を入力してください |
| 塗りマス 0 | 1マス以上塗ってください |
| デッキ上限（新規のみ） | デッキは最大 5 枚です |

**新規作成の確認ダイアログ**

- タイトル: 「カードを作成しますか？」
- 本文: 「このイメージと名前でカードを作成します。よろしいですか？」
- 確定: **作成する**（primary スタイル） / キャンセル

**編集時のデータ保持**

- `updateCardFromDrawing`：`id`・`name`・`attribute`・`rarity`・`stars`・`wins`・`losses`・`reviveCount`・`createdAt`・`canvasSize` は維持

## 5. カード作成仕様

### 5.1 入力

| 項目 | 必須 | 説明 |
|------|------|------|
| `name` | ○（新規のみ入力） | プレイヤーが付けるカード名（最大32文字）。**編集時も入力欄で変更可**（保存時課金） |
| `pixels` | ○ | `canvasSize` の正方形グリッド。色は **現在レベルで解放済みのパレット色** のみ有効。未塗りは `null` |
| `canvasSize` | ○（新規のみ選択） | 解放済みサイズから選択。**編集時は作成時の値を維持** |

### 5.2 色比率の算出

塗られたマスのみを対象とする（未塗りは分母に含めない。実装時にコメントで固定すること）。

```
R = 赤マス数 / 塗ったマス合計
W = 白マス数 / 塗ったマス合計
B = 黒マス数 / 塗ったマス合計
（R + W + B = 1）
```

### 5.3 シード（hash）

- **入力文字列**: `name` + 正規化した `pixels`（JSON 文字列など、実装で1形式に固定）
- **処理**:  cryptographic hash または十分な衝突耐性の hash → 数値化（例: 0〜1 の `hashAttack`, `hashDefense`）
- **性質（プロトタイプ）**: 同じ名前 + 同じ絵 → **常に同じ** 属性・BP（決定論）
- **性質（将来・AI）**: 作成時に1回だけ判定し、結果を `Card.attribute` に保存。以降は編集しても属性は変わらない

### 5.4 属性決定

#### 5.4.1 プロトタイプ現行（解放済み属性からランダム）

**方針**: 新規作成時、**ユーザーレベルで解放済みの全属性** から权重付きランダム抽選。実装: `src/card/rollAttribute.ts` → `createCardFromDrawing`。

| 項目 | 内容 |
|------|------|
| 候補 | `getUnlockedAttributes(userLevel)` |
| 基本权重 | 各属性 **1.0** |
| 直近解放ブースト | スケジュール上 **最後に解放された属性** のみ **1.1**（`ATTRIBUTE_ROLL_RECENT_BOOST`） |
| 色・hash | **属性決定には不使用**（BP 算出の hash + 密度は [§5.5](#55-bp-決定) のまま） |
| 編集画面 | 属性は **変更不可**（表示のみ） |
| 作成後の変更 | カード詳細の **属性リタッチ**（300px）/ **属性セレクト**（💎100）— [ATTRIBUTE_SPEC §3.3](./ATTRIBUTE_SPEC.md#33-作成後の属性変更) |

- **新規作成時のみ** 自動決定。編集保存では属性を変更しない（[§5.8](#58-新規作成と編集)）。

#### 5.4.2 将来方針（AI による漢字1文字属性）

カード作成ボタン確定時（**新規作成のみ**）に、以下を入力として AI（またはサーバー経由の同等ロジック）へ渡し、**漢字1文字**の属性を返させる。表示は `ATTRIBUTE_META`（`src/config/attributes.ts`）の `label` に対応させる。

**入力（コスト最小化を想定）**

| 項目 | 内容 |
|------|------|
| カード名 | プレイヤー入力 |
| イメージの見た目 | ASCII ピクセル表現などコンパクトな形式（将来 Vision API 併用も可） |
| 色数 | 使用した色の種類数 |
| カラーバランス | 各色の比率・色調の雰囲気 |
| 塗りピクセル数・密度 | 塗ったマス数、`canvasSize` に対する密度 |
| キャンバスサイズ | `canvasSize` |
| 候補属性 | **作成時のユーザーレベルで解放済み**の属性のみ（漢字リスト） |

**方針**

- 色合いのバランス・イメージの意味・カード名を総合して属性を決める（色だけでは決めない）
- 結果はカードに保存し、**編集では再判定しない**
- API 失敗時はプロトタイプと同様の hash + 色比率フォールバックを検討
- **色のレベル解放と属性の追加**: 色を Lv N で解放し、属性を Lv N+1 で追加する案を検討中（詳細は実装時に確定）

### 5.5 BP 決定

- **方針**: `hash` + **密度**（塗ったマス / グリッド総マス数）で、属性ごとの **BP レンジ** 内の値を決定。
- 密度が高いほどレンジの **上限側** に寄せる（線形補間。`src/config/balance.ts` の `computeCardBaseBp`）。
- **レンジの中心**は **ユーザーレベル** と属性でスケール（`getUserBaseBp`）。

| タイミング | BP の算出 |
|------------|-----------|
| **新規作成** | 作成時の `userLevel` でレンジを決定。`bpBlend`（hash + 密度）でレンジ内の値を決める |
| **編集保存** | **保存時の `userLevel`** でレンジを再スケール。`attribute`・`rarity` は固定。`bpBlend` は絵の変更で変わりうる |

- レベルアップ後に既存カードを編集保存すると、同じ絵でも BP レンジが伸びる。絵を直すと `bpBlend` も変化しうる。
- プロトタイプ用の具体レンジ（例、要プレイテスト）:

| 属性 | BP レンジ（案） |
|------|-----------------|
| attack | 70 〜 100 |
| defense | 55 〜 85 |

※上記数値は指示書時点の仮値。`src/config/balance.ts` で一元管理すること。

### 5.7 キャンバスサイズとレベル解放

**方針**: 解放は **最大一辺長（px）の引き上げ**。新規作成時は **16 〜 現在の上限** の **任意整数** を **ドロップダウン** で選択（奇数も可）。編集時は作成時の `canvasSize` 固定（[§5.8](#58-新規作成と編集)）。

| 項目 | 内容 |
|------|------|
| 初期（Lv1） | 最大 **16** px |
| 解放間隔 | **L ≡ 3 (mod 5)** かつ L ≥ 8（**8, 13, 18, 23, 28, 33, 38, 43, 48**）のたび **+2 px** |
| Lv50 時点 | 最大 **34** px（`MAX_USER_LEVEL = 50` と対応） |
| Lv51+ | [将来拡張](#57-キャンバスサイズとレベル解放) — レベル上限引き上げに合わせて **+2px 帯を継続** |

**将来拡張（ユーザーレベル上限の引き上げ）**

### 5.8 新規作成と編集

新規作成と編集は、プレイヤーにとって別の目的を持つ。

| | **新規作成** | **編集** |
|--|--------------|----------|
| 目的 | 名前・属性・サイズを含め **ゼロから** 新しいカードを得る | 同じカードの **イメージを仕上げる**・名前変更・キャンバス拡大 |
| イメージ | 自由に描く | 修正可 |
| カード名 | 入力して決定 | **常時編集可** — 保存時にリネーム課金（**200px/回**。[ECONOMY §8.2](./ECONOMY_SPEC.md#82-カード名変更編集画面)） |
| 属性 | 作成時 **ランダム抽選**（§5.4.1） | 編集画面では **変更不可**。詳細の **リタッチ/セレクト** で変更可（§4.3.1） |
| キャンバスサイズ | 解放済みから選択 | **拡大のみ**（`calcCanvasUpgradeCost` = 新²−旧² px、基準は編集開始時サイズ）。保存前に元サイズへ戻せば **0**。中間サイズ変更時は一度作成時サイズへ戻してから拡大モーダル |
| 色 | 現在レベルの解放色 | 現在レベルの解放色（レベルアップ後に新色で彩りを足せる） |
| BP | 作成時 `userLevel` | **保存時 `userLevel`** で再計算 |
| レア | 作成時抽選 | **維持**（編集では再抽選しない） |
| 戦績 | 0 で開始 | **維持** |

**プレイヤー動機（設計意図）**

- **編集**: レベルアップで増えた色で印象を変え、同じカードを育てる。名前も保存時に変更可。BP はレベルアップ後の保存で伸びる。
- **新規作成**: 別の名前・**ランダム属性**・より大きいキャンバスなど、ゼロからの新カード向き。
- **属性変更**: 詳細の **属性リタッチ**（px）で運試し、**属性セレクト**（💎）で確定狙い。

**実装**

- 新規: `createCardFromDrawing`（`rollAttribute`）
- 編集: `updateCardFromDrawing`（`src/card/createCard.ts`）
- 保存課金: `EditorScreen` + `App.tsx`（`updateCard` + `calcEditorSaveCharges`）
- 属性変更: `attributeChange.ts` / `DeckCardDetailOverlay`
- キャンバス拡大: `EditorScreen` + `calcCanvasUpgradeCost`（`src/config/economy.ts`）

### 6.2 保存形式（Q18-C）

**永続化する（`localStorage` キー `dot5-battle-save-v1`）**

- **型の正**: `src/types/index.ts` の `SaveData` / `Card` / `UserProfile` 等。
- **`SAVE_SCHEMA_VERSION = 1`**（`src/storage/index.ts`）。旧 `deck[]` 単一デッキや欠損フィールドは読み込み時に正規化。
- 最小例（参考）:

```typescript
interface Card {
  id: string;
  name: string;
  pixels: PixelGrid;
  canvasSize: number;
  attribute: Attribute;
  bp: number;
  rarity: CardRarity;
  stars: 0 | 1 | 2 | 3;
  status: "active" | "lost";
  wins: number;
  losses: number;
  reviveCount: number;
  talismanEquipped: boolean;
  userNote?: string;
  createdAt: string;
}

interface SaveData {
  schemaVersion: number;       // 常に 1
  user: UserProfile | null;
  economy: UserEconomy;
  inventory: UserInventory;
  adState: AdState;
  decks: DeckLayout[];         // 最大5スロット × 各5枚

  missionState: MissionState;
  // subscription, shopPurchase, battleHistory, paletteShopUnlocks 等
}
```

**ユーザー設定（オーディオ・案内）**

| フィールド | 説明 |
|------------|------|
| `soundEnabled?` | BGM ON/OFF（省略時 **false**）。`normalizeSoundEnabled` |
| `deckIntroSeen?` | マイデッキ初回案内（`DeckIntroModal`）を表示済みか |
| `shopPurchase.mockLifetimeSpendYen?` | **DEV テスト用** — モック現金課金累計（円） |
| `adState.mockAdsWatchedTotal?` | **DEV テスト用** — モック広告視聴完了累計 |

**EXP・レベルアップ（`src/user/level.ts`）**

- 1 戦の獲得 EXP = **現在レベル + 倒した敵カード枚数**（0〜5。勝敗不問）+ **上振れボーナス**（勝利・相手デッキ戦力が高いときのみ、最大 +3）
- 上振れ: 相手戦力 / 自分戦力 が **≥1.1 → +1**, **≥1.2 → +2**, **≥1.3 → +3**（参考。ゲーム結果への直接影響は EXP のみ）
- レベル L → L+1 に必要な EXP = **`EXP_LEVEL_BASE + L`**（初期 `BASE=10` → Lv.1→2 は 11）
- 累計 EXP 到達点: `totalExp(L) = (L-1)*BASE + (L-1)*L/2`
- 上限 **`MAX_USER_LEVEL = 50`**。降格なし
- UI: プログレスバーのみ（`expInLevel / expToNextLevel`）。レベルアップ時バー長は不変でレンジ切替

**デッキ戦力（`src/card/power.ts`・参考表示のみ）**

- カード戦力 = `round(BP × bpWeight[属性] + flatBonus[属性])`
- デッキ戦力 = 5枚のカード戦力合計（編成補正なし）
- Lv0: 攻撃 `{ bpWeight: 1, flatBonus: 0 }` / 防御 `{ bpWeight: 1, flatBonus: 20 }`（盾能力の反映）
- マイデッキ画面サブ情報に **戦力 N** を表示。バトル結果・ダメージ計算には使わない

```typescript
interface UserAuth {
  email: string;
  // password はクライアントに保持しない
}
```

- 現在 BP
- 盾の付与状態・使用済みフラグ

| 追加 | **新規カード作成** → イメージ作成画面。上限 5 枚 |
| 編集 | カード行タップ → カード編集画面。**イメージ**・**名前**・**キャンバス拡大** が変更可。属性は詳細のリタッチ/セレクト。保存で `pixels`・`bp`・名前を更新（[§5.8](#58-新規作成と編集)） |

## 7. 戦闘前フロー

`BattleSetupScreen` は **matching → reveal → setup → battle** の4フェーズ（履歴再戦は matching を省略 — [§11.4](#114-バトル履歴再戦)）。

### 7.0 通常 CPU 戦 — マッチング・reveal

| フェーズ | 内容 |
|----------|------|
| **matching** | 相手探索演出（**2〜4秒** ランダム、`rollMatchingDurationMs`）。敵スロットは空、オーバーレイ「対戦相手を探しています」。**キャンセル不可** |
| **reveal** | マッチング完了後、敵5枚をスロットに **即表示**。中央に **3秒カウントダウン**（`MATCH_REVEAL_COUNTDOWN_SEC`）。ラベル「マッチング完了、バトル準備まで残り N 秒」。**キャンセル** 可（**25px**、`BATTLE_MATCH_CANCEL_COST`） |
| **setup** | 従来の配置フェーズ（§7.1） |
| **battle** | バトル本編 |

- BGM: `battleSetup` 画面で **Battle.mp3** が再生開始（[§4.10](#410-サウンドbgm)）。
- 実装: `enableOpponentMatching={true}`（`App.tsx` — 通常 CPU 戦のみ）。

### 7.1 配置と操作（setup フェーズ）

1. プレイヤーの **5枚** と CPU の **5枚** を盤面スロットに表示する。
2. スマホ縦持ちを最優先し、中央盤面に敵・自分それぞれ **前衛2スロット / 後衛3スロット** を表示する。
3. **setup 開始時**、プレイヤー5枚は `randomFormation` で前衛・後衛に **ランダム配置済み**。ユーザーはタップで **スロット間の入れ替え** のみ行う（手札からの配置UIは廃止）。
4. **通常 CPU 戦**: reveal 完了時点で敵5枚はスロットに表示済み。setup 開始時に必要なら CPU 配置アニメ。
5. 準備時間は **30秒**。setup 画面表示と同時にカウント開始（プレイヤー陣営に残り秒数を表示）。
6. 画面下部ボタンは常に **「準備完了」**。押すとバトル開始。
7. 30秒経過時は、**その時点の配置のまま** バトル開始（「準備完了」未押下でも開始）。
8. バトル開始時、ガイダンスは **「カードオープン」** 等の既存演出に続く。

### 7.2 配置ルール

| 項目 | ルール |
|------|--------|
| 初期配置 | 前衛2枚、後衛3枚（5枚ランダム配置済みで setup 開始） |
| 自分の盤面向き | 前衛2枚が中央寄り、後衛3枚が手前 |
| 敵の盤面向き | 前衛2枚が中央寄り、後衛3枚が奥 |
| 操作方式 | ドラッグではなくタップ方式（スロット間の入れ替え） |
| 選択解除 | 選択中の同じカードを再タップ |
| スロット → スロット | 入れ替え |
| カウントダウン | プレイヤー陣営に残り秒数を表示 |
| スロット枠 | カードありスロットの枠は非表示 |
| 画面タイトル | 「バトル準備」「バトル！」の画面上部タイトルは表示しない |

### 7.2 用語

| 用語 | 意味 |
|------|------|
| 行動カード | そのターンに自分が選んで行動させる1枚 |
| 前衛 | 近接攻撃できる2スロット |
| 後衛 | 通常の近接攻撃はできない3スロット |
| 墓地 | 撃破されたカードを小さく並べる表示領域 |

## 8. 戦闘ルール

### 8.1 試合の流れ

- **1ターン** を繰り返す。
- **終了条件**: どちらか一方の **5枚すべてが BP 0**。
- **勝利**: 相手の5枚すべて撃破。
- **敗北**: 自分の5枚すべて撃破。
- 撃破されたカードは盤面スロットから外れ、墓地へ移動する。元スロットは空きスロットになる。

### 8.2 ターン開始：行動カード選択

| ルール | 内容 |
|--------|------|
| 選択枚数 | 1ターンに行動できるカードは各陣営 **1枚だけ**（**行動カード**） |
| プレイヤー / CPU | 行動可能なカードから1枚を選ぶ |
| 同時解決 | 双方の選択を揃えたうえで [ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#6-ターン処理順) の順に解決 |
| 剣・前衛 | 近接攻撃（敵前衛） |
| 盾・前衛 | 近接 **または** 盾付与（§9 / ATTRIBUTE_SPEC） |
| 凍結 | 凍結中のカードは **行動カードに選べない** |

属性ごとの行動可否・対象は **[ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)** を正とする。

### 8.3 攻撃対象（概要）

| 攻撃種別 | 対象 |
|----------|------|
| 近接攻撃 | 敵前衛（剣・盾・力・両・毒・氷・前衛の弓 等） |
| 弓 | 敵前衛・後衛（忍ステルス除く）。与ダメ currentBp **100%**・無反撃（ATTRIBUTE_SPEC §4.4） |
| 嵐 | 生存敵からランダム2体（別ユニット）。与ダメ currentBp **50%**・無反撃（ATTRIBUTE_SPEC §4.9） |

### 8.4 ターン処理：フェーズ

詳細・数値は **[ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#6-ターン処理順)** を参照。

```
【ターン開始 — 行動選択の前】
① ターン見出し・毒 DoT（§5.1）
② 撃破処理（DoT 起因）
③ 前衛補充（DoT 起因。プレイヤーは補充必須）

【行動確定後】
④ 行動カード選択（各陣営1枚）※ UI 上は ①〜③ の後
⑤ 癒（選ばれた側・currentBp 順）
⑥ 照（選ばれた側・currentBp 順）
⑦ 盾付与（選ばれた側）
⑧ 攻撃（弓・近接・両・嵐 等）
⑨ 撃破・墓地移動
⑩ 前衛補充（戦闘起因。弓が前衛に上がっても BP は変化しない）
```

※ 詳細・番号対応は [ATTRIBUTE_SPEC §6](./ATTRIBUTE_SPEC.md#61-1ターン全体の流れ) を正とする。

### 8.5 前衛補充

配置を以下のように表す。

```
前衛: A B
後衛: C D E
```

| 空き前衛 | 上がれる後衛 |
|----------|--------------|
| A（左前衛） | C（左後衛）または D（中央後衛） |
| B（右前衛） | D（中央後衛）または E（右後衛） |

- D は左右どちらにも上がれる共有候補。
- 片側だけ空いている場合、対応候補がいなければ空きのまま次ターンへ進む。
- 両前衛が空いた場合も同じ対応ルールを守る。
- 複数候補がある場合、プレイヤーは補充順と移動先を自由に選べる。
- 候補が **1枚だけ** の場合は自動で前衛へ補充する（ユーザー操作不要）。
- 候補が **2枚** の場合のみ、ユーザーに選択を促す。
- CPU は対応候補からランダムに上げる。
- 補充操作はタップ方式: 上げる後衛カードをタップ → 同じカードを再タップで確定（1回目で選択、2回目で確定）。
- ガイダンス文言:
  - 未選択時: **「前衛に移動する後衛カードを選択」**
  - 選択後: **「もう一度タップで確定」**

### 8.6 撃破後

- BP 0 のカードはその試合では **もう選択できない**。
- 撃破カードは墓地に小さく表示し、グレーアウトまたは撃破ビジュアルを付ける。
- 自分と敵の墓地は分けて表示する。
- 盤面レイアウトは、準備中・バトル中・墓地移動後も固定する。

### 8.7 試合終了表示

| 項目 | 内容 |
|------|------|
| 中央ガイダンス | `TURN N：バトル終了！`（勝敗文言は出さない） |
| 敵陣営エリア | 墓地＋敵スロット全体の中央に **WIN** または **LOSE** を大きく重ね表示 |
| 味方陣営エリア | 味方スロット＋墓地全体の中央に **WIN** または **LOSE** を大きく重ね表示 |
| スタイル | グローは使わず、鮮やかな塗り＋太い縁取り（WIN＝黄、LOSE＝赤系） |
| 対人戦（将来） | 各クライアントで自分の陣営側に WIN/LOSE を表示。中央は双方共通の終了メッセージ |
| 終了後操作 | `デッキ` / `新規バトル` を画面下端に固定。盤面本体はスクロール可能 |

### 8.8 攻撃解決の詳細

#### 近接・反撃（剣/盾 Lv0 ＋ 全属性共通）

- **原則**: 攻撃ダメージは **currentBp** 基準（[ATTRIBUTE_SPEC §2.2](./ATTRIBUTE_SPEC.md#22-ダメージの基準)）。
- **相打ち**（同一2枚が互いに近接の行動カード）: 双方 **100%** の相互ダメージ。
- **一方的**: 攻撃側 **100%**、反撃 **50%**。
- 詳細・弓/両/嵐/毒/氷/忍は **[ATTRIBUTE_SPEC.md](./ATTRIBUTE_SPEC.md)**。

#### 盾の相互作用

| 状況 | 結果 |
|------|------|
| 防御側のみ盾あり | 防御側はダメージ0、盾消費。攻撃側は通常通り反撃ダメージ |
| 攻撃側のみ盾あり | 攻撃側は反撃ダメージ0、盾消費。防御側は通常通りダメージ |
| 双方とも盾あり | 双方の盾が消え、双方BPは変わらない |

#### 複数攻撃の処理順

同一ターンに複数の近接攻撃が発生する場合:

1. 互いに同じ2枚のカードが攻撃し合う場合は **1戦闘** としてまとめる（両矢印）。
2. 残りの戦闘は **1件ずつ順番** に処理する。
3. 戦闘の優先順位は **攻撃を仕掛けたカードのBPが高い順**。
4. 同率なら双方の最大BPが高い順 → さらに同率なら防御側BPが高い順。

UI 演出もこの処理順に従い、各戦闘ごとに「ダメージ表示 → BPカウントダウン」を行う。

#### 複数攻撃の処理順

1. 互いに同じ2枚が近接の行動カード → **1戦闘**（相打ち / `bidirectional`）。
2. 残りは **1件ずつ** 処理。
3. 優先: **行動カードの currentBp 高い順**（同率は [ATTRIBUTE_SPEC §7 tie-break](./ATTRIBUTE_SPEC.md#7-同率時の解決順tie-break)）。

UI 演出もこの順に従う。

## 10. CPU 仕様

### 10.1 配置

- CPU の5枚配置は、プロトタイプではランダム。
- 準備中は上部の敵手札に裏面5枚を表示し、空スロットへ1枚ずつ飛ばして配置する。
- 配置順は **前衛左 → 前衛右 → 後衛左 → 後衛中央 → 後衛右**。
- バトル開始時に同順で1枚ずつ表面へ反転する。

### 10.2 行動選択

- 初期実装では、CPUの通常ターン行動はランダムでよい。
- 実行可能な行動候補を列挙し、その中からランダムに1つ選ぶ。
- 攻撃行動の場合、敵前衛の生存カードからランダムに対象を選ぶ。
- 盾付与行動の場合、盾を持っていない生存味方からランダムに対象を選ぶ（**付与者自身は除く**）。
- 将来は、前衛保護・低BP保護・撃破可能攻撃などを評価するスコア式AIへ差し替えてよい。

### 10.3 前衛補充

- 空いた前衛に対して、対応候補からランダムに上げる。
- 候補がいなければ補充しない。

### 10.4 CPU デッキ

- プロトタイプ: 生成ルールまたは `src/data/cpu-decks.json` 等の **固定デッキ** で可。
- 将来: 他ユーザーの5枚メタデータを取得し、同じルールで CPU が操作。

## 12. 実装構成の指針

### 12.1 戦闘エンジン API（案）

```typescript
// 例: 純関数。UI は state を渡し、結果 state を受け取る
function resolveTurn(state: BattleState, choices: TurnChoices): ResolveTurnResult;
function promoteUnit(state: BattleState, side: BattleSide, from: BoardPosition, to: BoardPosition): BattleState;
function isBattleOver(state: BattleState): "player" | "cpu" | null;
```

- `TurnChoices`: 行動カード、行動種別、攻撃対象、盾対象等
- ユニットテストを `game/*.test.ts` に書き、Unity 移植前の仕様固定に使う

### 12.2 戦闘状態・イベントログ（案）

```typescript
type BoardPosition =
  | "frontLeft"
  | "frontRight"
  | "backLeft"
  | "backCenter"
  | "backRight";

type BattleActionType = "meleeAttack" | "grantShield";

interface BattleFormation {
  slots: Record<BoardPosition, BattleUnit | null>;
  defeated: BattleUnit[];
}

interface BattleEvent {
  type:
    | "action_selected"
    | "shield_granted"
    | "attack"
    | "blocked"
    | "defeated"
    | "promoted";
}
```

- バトルログは通常画面には表示しない。
- リプレイ機能やお気に入り保存に備え、内部的にはログを維持する。
- 可能なら文字列ログだけでなく、演出・リプレイに使える構造化イベントログを保持する。
- 倒れたカードは `defeated` へ移動し、元スロットは `null` にする。

### 12.3 カード生成 API（案）

```typescript
function createCardFromDrawing(name: string, pixels: PixelGrid): Card;
function updateCardFromDrawing(existing: Card, name: string, pixels: PixelGrid): Card;
```

- `createCardFromDrawing`：新規 `id`・戦績 0 で生成。`attribute`・`rarity`・`bp` を算出
- `updateCardFromDrawing`：既存の `id` / `name` / `attribute` / `rarity` / `stars` / `wins` / `losses` / `reviveCount` / `createdAt` / `canvasSize` を保持し、`pixels` と `bp` のみ更新（`bp` は保存時 `userLevel` で再算出）

### 12.4 確認ダイアログ（共通）

| 用途 | confirmVariant | 確定ラベル例 |
|------|----------------|--------------|
| カード削除 | `danger`（既定） | 削除する |
| カード作成 | `primary` | 作成する |

### 12.6 画面・演出指針

#### レイアウト

- スマホ縦持ちを最優先レイアウトとする。
- PC では body 背景の上に **phone シェル** として中央配置する（`--phone-shell-w: 26.5rem`、バトル画面は `--phone-shell-battle-w: 22rem`）。
- 盤面レイアウトは準備中・バトル中・墓地移動後も固定する。
- 手札・配置スロット・バトル中・墓地の **すべてのカード** は、同一の内部レイアウト（パーツ位置・サイズ・マージン）で表示する。
- カードサイズは `--formation-card-w` / `--formation-card-h` で統一する。
- 中央はガイダンス表示と矢印・ダメージ演出のために使う。

#### イメージ作成 / カード編集画面

- 画面上部は共通ブランドタイトルのみ。**画面切替ナビは非表示**。
- パレット → キャンバス → カード名 → 説明文の間に **約 0.7〜0.85rem** の縦余白を確保する。
- フッター（**カード作成** / **保存**、**マイデッキに戻る**）は画面下部に固定配置する。

#### 戦闘準備・戦闘画面

- 戦闘準備・戦闘中は画面上部の「バトル準備」「バトル！」タイトルおよびアプリ共通ヘッダを **非表示** とする。
- 中央ガイダンスは **1行帯**（ピンク〜紫グラデーション、左右アクセントライン）。`TURN N：操作ヒント` 形式で常に1行。
- 準備フェーズは盤面を `formation-setup-body` でスクロール可能にし、下部の **準備完了** は固定。
- バトル終了後は盤面を `formation-battle-body` でスクロール可能にし、`デッキ` / `新規バトル` を下端固定。
- スロット枠は配置中のみ点線表示。双方配置完了後およびバトル中は、カードありスロットの枠を非表示（`is-frameless`）。
- 試合終了時、中央は `バトル終了！`、各陣営エリア（`formation-field-cpu` / `formation-field-player`）に WIN/LOSE を重ねる。

#### ガイダンス・操作

- ガイダンス文言は短い日本語で統一する。
- ターン表示は `TURN 1` のような英字表記とする。
- 行動カード選択中は選択カードをハイライト表示する。
- 選択中の同じカードを再タップすると選択解除する（専用の「選択解除」ボタンは置かない）。
- 防御属性・前衛で攻撃と盾付与の両方が可能なとき、**行動ボタンは出さない**。敵前衛タップで攻撃、盾なし味方タップで盾付与。
- 攻撃対象選択時は **敵前衛** を赤系でハイライトする。盾付与対象は **味方** を緑系でハイライトする。

#### 盾・バフ表示

- 盾状態はカード全体を覆う半透明表示ではなく、**カード左上の小さなバフアイコン**（🛡）で示す。
- 将来のバフ・デバフも同じ左上エリアに並べられる構造とする。
- BP は右上にライトな色のバッジ表示とする。

#### ターン演出（攻撃・盾）

1. **盾付与フェーズ**（該当時）  
   緑系の矢印で付与元→付与先を表示する。

2. **攻撃フェーズ**（1戦闘ずつ順番に）  
   各戦闘について次の順で再生する。

   | サブフェーズ | 内容 |
   |--------------|------|
   | ダメージ表示 | 関係するカードの中央に `−85` などのダメージ数値を大きく表示。盾で0の場合は非表示 |
   | BP変化 | ダメージ表示を消し、右上BPを **from→to** でカウントダウンアニメーション |
   | 次へ | 上記完了後、次の戦闘へ。全戦闘完了後にターン終了処理へ |

3. **矢印**
   - 攻撃は **カードの端から端** へ向ける（中央同士ではなく、矩形外周の交点を始点・終点とする）。
   - 相互攻撃の場合は両矢印（↔）表示。
   - 対象が異なる同時攻撃は片矢印2本を同時表示。
   - **BPカウントダウン中は攻撃矢印を非表示** にする（ダメージ表示フェーズでのみ表示）。
   - ダメージ表示は矢印レイヤーより上の専用レイヤーに描画し、矢印に隠れないようにする。

4. **その他**
   - 処理中のカードは揺れエフェクトを付ける。
   - バトルログは画面に出さない。内部ログ・構造化イベントはリプレイ用に維持する。

#### タイミング定数

| 定数名 | 値 (ms) | 説明 |
|--------|---------|------|
| `CLASH_MS.enter` | 530 | スロット→中央 |
| `CLASH_MS.impact` | 380 | 中央衝突 |
| `CLASH_MS.damage` | 500 | ダメージ数値表示時間 |
| `CLASH_MS.bp` | 600 | BPカウントダウン時間 |
| `CLASH_MS.exit` | 530 | ターン演出終了後の待機 |

#### resolveTurn 戻り値（攻撃演出用）

各 `AttackPlayback` には以下を含める。

- 攻撃側・防御側の位置（`fromSide` / `fromPosition` / `toSide` / `toPosition`）
- 防御側ダメージ・BP変化（`damage`, `bpFrom`, `bpTo`）
- 攻撃側ダメージ・BP変化（`attackerDamage`, `attackerBpFrom`, `attackerBpTo`）
- 相互攻撃フラグ（`bidirectional`）
- 処理後状態（`stateAfter`）
